# <a name="iactivescriptaddnameditem"></a>IActiveScript::AddNamedItem
Добавляет имя элемента корневого уровня пространства имен обработчика скриптов. Корневой элемент — это объект, с помощью свойств и методов, источник события или все три.  
  
## <a name="syntax"></a>Синтаксис  
  
```  
HRESULT AddNamedItem(  
    LPCOLESTR pstrName,  // address of item name  
    DWORD dwFlags          // item flags  
);  
```  
  
#### <a name="parameters"></a>Параметры  
 `pstrName`  
 [in] Адрес буфера, который содержит имя элемента, как видно из скрипта. Имя должно быть уникальным и является сохраняемым.  
  
 `dwFlags`  
 [in] Флаги, связанные с элементом. Может представлять собой сочетание этих значений:  
  
|Значение|Значение|  
|-----------|-------------|  
|SCRIPTITEM_CODEONLY|Указывает, что именованный элемент представляет объект только в коде и не имеет узла `IUnknown` нужно связать с этим объектом, только в коде. Основное приложение имеет только имя для этого объекта. В объектно ориентированных языках, таких как C++ этот флаг должен создать класс. Не все языки поддерживают этот флаг.|  
|SCRIPTITEM_GLOBALMEMBERS|Указывает, что элемент является коллекции глобальных свойств и методов, связанный со скриптом. Как правило, обработчик сценариев приводит к пропуску имени объекта (отличного от с целью его использования в качестве файла cookie для [IActiveScriptSite::GetItemInfo](../../winscript/reference/iactivescriptsite-getiteminfo.md) метод, или для разрешения явные области) и предоставлять его члены как глобальный переменные и методы. Это позволяет узлу в библиотеку (функции времени выполнения и т. д.) доступны для скрипта. Он остается на обработчик скриптов для работы с именем конфликтов (например, если два элемента SCRIPTITEM_GLOBALMEMBERS иметь методы с тем же именем), несмотря на то, что ошибка не должен возвращаться из-за этой ситуации.|  
|SCRIPTITEM_ISPERSISTENT|Указывает элемент должна сохранить при сохранении обработчика скриптов. Аналогичным образом, этот флаг установлен указывает, что переход состояния инициализации должны содержать элемента имя и сведения о типе (обработчика скриптов Однако выпустить все указатели на интерфейсы для фактического объекта).|  
|SCRIPTITEM_ISSOURCE|Указывает, что элемент источник событий, которые скрипт может приемника. Дочерние объекты (свойства объекта, сами по себе объекты) можно также источника событий в скрипт. Это не является рекурсивным, но он обеспечивает удобный механизм для обычной практикой, например, создания элементов управления контейнера и всех его элементов.|  
|SCRIPTITEM_ISVISIBLE|Указывает, доступны в пространстве имен скрипта для предоставления доступа к свойства, методы и события элемента, имени элемента. По соглашению свойства элемента содержит дочерние элементы данного элемента; Таким образом все свойства дочерних элементов объекта и методы (и их дочерние элементы, рекурсивно) будут доступны.|  
|SCRIPTITEM_NOCODE|Указывает, что элемент — это имя добавляемое пространство имен этого сценария и не следует обрабатывать как элемент, для которого должен быть связан код. Например без этот флаг устанавливается, VBScript будет создан отдельный модуль для именованного элемента и C++ может создать класс-оболочку отдельные именованного элемента.|  
  
## <a name="return-value"></a>Возвращаемое значение  
 Возвращает одно из следующих значений:  
  
|Возвращаемое значение|Значение|  
|------------------|-------------|  
|`S_OK`|Выполнено.|  
|`E_INVALIDARG`|Недопустимый аргумент.|  
|`E_POINTER`|Указан недопустимый указатель.|  
|`E_UNEXPECTED`|Вызов не ожидалось (например, обработчик сценариев еще не загрузки или инициализации).|
