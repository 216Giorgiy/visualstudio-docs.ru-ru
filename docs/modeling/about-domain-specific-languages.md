# О доменных языках
[!INCLUDE[vs2017banner](../code-quality/includes/vs2017banner.md)]

В отличие от языка, например c\# или общего назначения UML доменный язык \(DSL\) предназначен для выписки выражены в указанной области проблем или домен.  
  
 Включает известные DSLs регулярные выражения и SQL.  Каждый DSL гораздо лучше, чем язык общего назначения для описания операции в текстовых строках или базе данных, но очень плохо для описания идеи, за пределами своей области.  Отдельные отрасли также имеют свое собственное DSLs.  Например, в отрасли радиосвязей, языки описания вызова часто используются для определения последовательности состояний в телефонном вызове и в отрасли туризма воздуха стандартный DSL используется для описания резервирования coast.  
  
 Ваша компания и проект также специальными которых описывается с наборами понятий, описанных с DSL.  Например, можно указать DSL для одного из следующих приложений:  
  
-   План путей навигации веб\-сайта.  
  
-   Монтажные схемы для электронных блоков.  
  
-   Сети конвейерных лент и оборудования для обработки багажа авиапорта.  
  
 При проектировании DSL указывается a *доменный класс* для каждого из важных концепций в домене, например страницы, светильник или стол возврата авиапорта.  Указании *доменные связи* в качестве гиперссылки, сети или конвейерная лента для связывания понятия.  
  
 Пользователи данного DSL создают *модели.* Модели *экземпляры* DSL.  Например, они описывают заданного веб\-сайта или проводку конкретного устройства или системы управления багажа в указанном авиапорте.  
  
 Пользователи могут просматривать модель в виде схемы или окна формы.  Модели можно также просмотреть в виде XML, как они хранятся.  При указании DSL указывается как экземпляры каждого доменного класса и связи отображаются на экране пользователя.  Типичный DSL показано, как коллекция значков или прямоугольников, подключенных стрелками.

## Что можно делать с DSLs  
 Типичное приложение DSL создавать программный код или другие артефакты.  При указании можно указать свой DSL *текстовые шаблоны* чтение модели DSL, и создает текстовые файлы.  
  
 Например, можно написать шаблоны, которые принимают план авиапорта и создают часть программного обеспечения для обработки багажа, а также некоторые из документов пользователя, описывающие план.  
  
 После определения DSL, его можно распределять другим пользователям, которые могут устанавливать его на своих собственных компьютерах.  Пользователи данного DSL могут создавать и редактировать модели внутри [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)].  
  
 Также можно указать команды меню и другие средства, помогающие пользователям редактировать DSL, ограничения проверки обеспечить DSL используется правильно и шаблоны элементов, которые помогают пользователям создания новых экземпляров.  Можно создать один или несколько DSLs программу\-оболочку с их и другими средствами [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] расширения, как интегрированный пакет.  
  
 Обычно доменный язык создается, когда команда разработчиков должна выполняться аналогичный код для некоторых продуктах.  Например, компания, которая специализирует в системах управления багажа могут определить отслеживание DSL багажа, из которой они могут создать часть кода для каждой установки.  Преимущества DSL, что он может быть понят их клиентами, что код, созданный из него надежн, а система может быть быстро обновления, если требования клиентов.  
  
 [!INCLUDE[dsl](../modeling/includes/dsl_md.md)] позволяет создать доменный язык, который имеет собственный графический конструктор и собственную нотации схемы, а затем использовать язык для создания соответствующий исходный код для каждого проекта.  
  
## Разработка решений, обусловленных требованиями домена  
 Разработка решений, обусловленных требованиями домена \- это процесс определения частей приложений, которые могут быть моделируются с помощью доменного языка, и затем построения самого языка и развертывать ее разработчикам приложений.  Разработчики используют доменный язык для построения модели, характерные для их приложений, использующих модели для создания исходного кода, а затем использует исходный код для разработки приложений.  
  
## Аспекты графической разработки, обусловленных требованиями домена  
 Графический доменный язык должен включать следующие функции:  
  
-   Notation  
  
-   Модель домена  
  
-   Создание артефактов  
  
-   Сериализация  
  
-   Интеграция с Visual Studio  
  
### Notation  
 Доменный язык должен иметь достаточно небольшой набор элементов, которые могут быть легко указанным и расширенным для представления конструктора, обусловленных требованиями домена.  Нотация состоит из фигур, которые представляют элементы и соединителей, представляющие связи между элементами в графической поверхности схемы.  IN [!INCLUDE[dsl](../modeling/includes/dsl_md.md)]фигуры, могут быть расширены и уточнены представления элементов вашего доменного языка.  
  
### Модель домена  
 Доменный язык должен объединить набор элементов и отношений между ними в когерентную грамматику.  Необходимо также определить, является ли элементы и отношения сочетания являются допустимыми.  Например, языки программирования, как правило, препятствующие циклическое наследования, в которых классу, производные от второго класса и второй класс является производным от первого класса.  Ограничения также можно использовать, чтобы представить бизнес\-логику, например, одна продажа не может зависеть себя.  [!INCLUDE[dsl](../modeling/includes/dsl_md.md)] использует ограничения для выражения типы ограничений которых наиболее доменных языков.  
  
### Создание артефактов  
 Одна из основных целях доменного языка создать артефакт, например, исходный код, XML\-файл или некоторые другие годные к использованию сведения.  Как правило, изменения в модели означает изменение артефакте.  Можно использовать [!INCLUDE[dsl](../modeling/includes/dsl_md.md)] создать артефакты и повторно создать их при изменении модели.  
  
### Сериализация  
 Доменный язык должен сохранять в несколько форм, можно изменять, сохранять, закрыть и перезапуска.  [!INCLUDE[dsl](../modeling/includes/dsl_md.md)] используется xml\-формат, который позволяет задавать и настраивать способ сериализации или сохранен в доменный язык.  
  
### Интеграция с Visual Studio  
 Поскольку [!INCLUDE[dsl](../modeling/includes/dsl_md.md)] размещается в  [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)]он расширяет много  [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] окна и элементы управления.  Он также позволяет настраивать функциональности элементов команд меню, панели инструментов и других элементов пользовательского интерфейса.  
  
 Также можно создать адаптер шины модели для доменного языка.  Этот адаптер позволяет ссылаться на модели и элементами в модели, и позволяет писать код, который может получить доступ и обновление экземпляра DSL.  Это мощная с помощью механизма шины модели, можно написать [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] расширения, работающие с несколькими моделями.  Можно также написать отдельные приложения, работающие с моделями.  Дополнительные сведения см. в разделе [Интеграция моделей с помощью Visual Studio Modelbus](../modeling/integrating-models-by-using-visual-studio-modelbus.md).  
  
## Преимущества разработки, обусловленных требованиями домена  
 Доменный язык может предоставить следующие преимущества:  
  
-   Содержит конструкции, которые точно по размеру места проблемы.  
  
     В отличие от языки общего назначения, доменный язык состоит из элементов и отношений, которые непосредственно представляют логику пробелы проблемы.  Например, приложение должно включать элементы политики страхования для политик и утверждений.  Доменный язык упрощает проектирование приложения и выполняется поиск и исправление ошибки логики.  
  
-   Позволяет non\-разработчикам и продажи, которые не знают домен понимают общую компоновку.  
  
     С помощью графического доменного языка, можно создать визуальное представление домена, так что non\-разработчики могут легко понимают конструкцию приложения.  
  
-   Упрощает создание заполнитель последнего приложения.  
  
     Разработчики могут использовать код, их модель формирует для создания приложения заполнителя, они могут отображать клиентам.  
  
## Процесс разработки, обусловленных требованиями домена  
 Большинство группы разработки программного обеспечения, использующих доменных языков, выполните следующие шаги, чтобы создать и использовать их моделей.  
  
-   Команда отличает переменные части домена из частей, которые никогда не изменяются.  
  
-   Разработчики пишут код для фиксированных частей и оставляет точки расширения для переменных частей.  
  
-   Ведущий разработчик программного обеспечения или архитектор создать доменный язык, который включает шаблоны разработки фиксированных частей домена и точек расширения для переменных частей.  
  
-   Ведущий разработчик программного обеспечения или архитектор развертывают доменный язык для разработчиков различных приложений, которые команда формирует.  
  
-   Каждый разработчик создает модели, которая применяется к конкретному приложению.
