# <a name="getting-started-with-domain-specific-languages"></a>Начало работы с доменными языками
В этом разделе объясняются основные понятия в определении и использовании доменного языка (DSL), созданные с помощью пакета SDK моделирования для Visual Studio.  


[!INCLUDE[modeling_sdk_info](includes/modeling_sdk_info.md)]

  
 Если вы не знакомы с DSL, рекомендуется при работе через **лаборатории средств DSL**, который можно найти на этом сайте: [пакет и пакет SDK моделирования](http://go.microsoft.com/fwlink/?LinkID=186128)  
  
## <a name="what-can-you-do-with-a-domain-specific-language"></a>Что можно сделать с помощью доменного языка  
 Доменный язык представляет собой нотацию, обычно графическом, предназначенный для определенной цели. Напротив языки, такие как UML общего назначения. В DSL можно определить типы элементов модели и связи между ними и как они должны быть представлены на экране.  
  
 При созданной DSL можно распределить его как часть пакета расширение интеграции Visual Studio (VSIX). Пользователи работают с DSL в [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)]:  
  
 ![Обозреватель, панель элементов и схема семейного дерева](~/docs/modeling/media/familyt_instance.png "FamilyT_Instance")  
  
 Нотация является частью DSL. Вместе со нотации пакет VSIX включает средства, которые пользователи могут применять для изменения и Создание материалов на основе их моделей.  
  
 Один из основного приложения доменных языков — создавать программный код, файлы конфигурации и другие артефакты. Особенно в крупных проектах и линии товаров, где будут созданы несколько вариантов продукта, создании многие аспекты переменной из DSL можно предоставить значительное увеличение надежности и очень быстрого реагирования на изменения требований.  
  
 Остальная часть этого обзора — руководство, в котором представлены основные операции создания и использования доменного языка в [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)].

## <a name="creating-a-dsl-solution"></a>Создание решения DSL  
 Для создания нового доменного языка, создайте новый [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] решения с помощью шаблона проекта доменного языка.  
  
#### <a name="to-create-a-dsl-solution"></a>Создание решения DSL  
  
1.  В меню **Файл** выберите пункт **Создать**, а затем команду **Проект**.  
  
2.  В разделе **типы проектов**, разверните **другие типы проектов** и выберите команду **расширения**.  
  
3.  Щелкните **конструктора доменного языка**.  
  
     ![Создание диалогового окна DSL](~/docs/modeling/media/create_dsldialog.png "Create_DSLDialog")  
  
4.  В **имя** введите **FamilyTree**. Нажмите кнопку **ОК**.  
  
     **Мастер доменного языка** открывает и отображает список шаблонов решений DSL.  
  
     Щелкните для просмотра описания каждого шаблона  
  
     Шаблоны полезны отправная точка. Каждая из них предоставляет полный рабочий DSL, который можно изменить в соответствии с потребностями. Обычно выбирается шаблон ближайшего вы хотите создать.  
  
5.  Для этого пошагового руководства выберите **минимальный язык** шаблона.  
  
6.  Введите расширение имени файла для DSL на соответствующей странице мастера. Это расширение будут использовать файлы, содержащие экземпляры DSL.  
  
    -   Выберите расширение, которое не связано с любого приложения на компьютере или на любом компьютере, где вы хотите установить DSL. Например **docx** и **htm** использовать нельзя расширений.  
  
    -   Мастер предупредит, если расширение уже используется в качестве DSL. В этом случае выберите другое расширение имени файла. Можно также сбросить экземпляр экспериментального пакета SDK для Visual Studio, чтобы удалить старые экспериментальные конструкторы. Щелкните **запустить**, нажмите кнопку **все программы**, **Microsoft Visual Studio 2010 SDK**, **средства**, а затем **Сброс Microsoft Visual Studio 2010 экспериментальный экземпляр**.  
  
7.  Проверьте другие страницы, а затем нажмите кнопку **Готово**.  
  
     Создается решение, которое содержит два проекта. Они называются Dsl и DslPackage. Файл схемы откроется именно именованных DslDefinition.dsl.  
  
    > [!NOTE]
    >  Большая часть кода, который можно увидеть в папках в двух проектах создается на основе DslDefinition.dsl. По этой причине большинство доменного языка изменений в этот файл.  
  
 После этого пользовательский интерфейс примет следующий вид:  
  
 ![Конструктор DSL](../modeling/media/dsl_designer.png "dsl_designer")  
  
 Данное решение определяет доменный язык. Дополнительные сведения см. в разделе [Обзор пользовательского интерфейса средств доменного языка](../modeling/overview-of-the-domain-specific-language-tools-user-interface.md).  
  
## <a name="the-important-parts-of-the-dsl-solution"></a>Важные части решения DSL  
 Обратите внимание, следующие аспекты новое решение.  
  
-   **Dsl\DslDefinition.DSL** это файл, который вы видите при создании решения DSL. Практически весь код в решении создается из этого файла, и большинство изменений, вносимых в определение DSL становятся здесь. Дополнительные сведения см. в разделе Работа с [работа со схемой определений доменных ЯЗЫКОВ](../modeling/working-with-the-dsl-definition-diagram.md).  
  
-   **Проект DSL** этот проект содержит код, определяющий доменный язык.  
  
-   **Проект DslPackage** этот проект содержит код, который позволяет экземпляры DSL, чтобы открывать и изменять в [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)].  
  
##  <a name="a-namedebugginga-running-the-dsl"></a><a name="Debugging"></a>Выполнение DSL  
 Можно запустить решение DSL, сразу после его создания. Позже можно изменить определения DSL постепенно, решение снова запустить после каждого изменения.  
  
#### <a name="to-experiment-with-the-dsl"></a>Чтобы поэкспериментировать с DSL  
  
1.  Щелкните **преобразовать все шаблоны** в панели инструментов обозревателя решений. Будет повторно создан большая часть исходного кода из DslDefinition.dsl.  
  
    > [!NOTE]
    >  При каждом изменении DslDefinition.dsl, необходимо нажать кнопку **преобразовать все шаблоны** перед Перестройте решение. Этот шаг можно автоматизировать. Дополнительные сведения см. в разделе [как автоматизировать преобразовать все шаблоны](http://msdn.microsoft.com/en-us/b63cfe20-fe5e-47cc-9506-59b29bca768a).  
  
2.  Нажмите клавишу F5, или в **отладки** меню, щелкните **начать отладку**.  
  
     Строит DSL и является установлены в экспериментальном экземпляре [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)].  
  
     Запустится экспериментальный экземпляр [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)]. Экспериментальный экземпляр использует ее параметры из отдельной ветви реестра, где [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] зарегистрированных расширений для целей отладки. Обычный экземпляров [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] не имеют доступа к расширениям зарегистрированы.  
  
3.  В экспериментальном экземпляре [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)], откройте файл модели с именем **тест** из **обозревателе решений**.  
  
     \- или -  
  
     Отладка проекта щелкните правой кнопкой мыши **добавить**, а затем нажмите кнопку **элемент**. В **добавить элемент** выберите тип файла DSL.  
  
     Файл модели открывается как пустая диаграмма.  
  
     Панели инструментов открывает и отображает средства, подходящие для типа диаграммы.  
  
4.  Используйте средства для создания фигур и соединителей на схеме.  
  
    1.  Для создания фигур, перетащите из средства пример фигуры на схему.  
  
    2.  Для соединения двух форм, щелкните инструмент соединителя пример, щелкните первую фигуру и выберите второй фигуру.  
  
5.  Щелкните метку фигур для их изменения.  
  
 Экспериментальный экземпляр [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] будет иметь следующий вид:

### <a name="the-content-of-a-model"></a>Содержимое модели  
 Содержимое файла, который является экземпляром DSL называется *модели*. Модель содержит *модель**элементы* и *ссылки* между элементами. Определение DSL определяет, какие типы элементов модели и могут существовать связи в модели. Например в DSL, созданная из шаблона минимальный язык, имеется один тип элемента модели и один тип связи.  
  
 Определение DSL можно указать, как модель отображается на схеме. Можно выбрать из множества стилей фигур и соединителей. Можно указать, что некоторые фигуры отображаются внутри других фигур.  
  
 Модель можно просмотреть в виде дерева в **Explorer** просмотра при редактировании модели. При добавлении фигуры в схеме элементы модели также появляются в обозревателе. Обозреватель может использоваться, даже если диаграммы не.  
  
 Если обозреватель в экземпляр отладки не виден [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)]на **представление** , выберите в меню **другие окна**и нажмите кнопку * \<ваш язык настроек* **Explorer**.  
  
### <a name="the-api-of-your-dsl"></a>API доменного языка  
 DSL создает API, который дает возможность чтения и обновления моделей, которые являются экземплярами DSL. Одно приложение API является создание текстовых файлов из модели. Дополнительные сведения см. в разделе [создание кода во время разработки с помощью текстовых шаблонов T4](../modeling/design-time-code-generation-by-using-t4-text-templates.md).  
  
 Отладка решения откройте файл шаблона с расширением «.tt». В этих примерах показано, как можно Создание текста из моделей и позволяют тестировать API DSL. Один из примеров записывается [!INCLUDE[vbprvb](../code-quality/includes/vbprvb_md.md)], другие в [!INCLUDE[csprcs](../data-tools/includes/csprcs_md.md)].  
  
 В каждом шаблоне файл — это файл, который создает. Разверните узел файла шаблона в обозревателе решений и откройте созданный файл.  
  
 Файл шаблона содержит краткое сегмент кода, содержащий все элементы в модели.  
  
 Созданный файл содержит результат.  
  
 При изменении файла модели, вы увидите соответствующие изменения в файлы, созданные после повторного создания файлов.  
  
##### <a name="to-regenerate-text-files-after-you-change-the-model-file"></a>Чтобы повторно создать текстовые файлы, после изменения файла модели  
  
1.  В экспериментальном экземпляре [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)], сохраните файл модели.  
  
2.  Убедитесь, что параметр имени файла в каждый tt-файл имеется ссылка на файл модели, используемой для экспериментов. Сохраните tt-файл.  
  
3.  Щелкните **преобразовать все шаблоны** на панели инструментов **обозревателе решений**.  
  
     \- или -  
  
     Щелкните правой кнопкой мыши шаблоны, которые вы хотите восстановить и нажмите кнопку **пользовательское средство**.  
  
 Можно добавить любое количество файлов текстового шаблона в проект. Каждый шаблон создает один файл результатов.  
  
> [!NOTE]
>  При изменении определения DSL в коде шаблона текста не будет работать, пока разработчик не обновит его.  
  
 Дополнительные сведения см. в разделе [создание кода из доменного языка](../modeling/generating-code-from-a-domain-specific-language.md) и [написание кода для настройки доменного языка](../modeling/writing-code-to-customise-a-domain-specific-language.md).  
  
## <a name="customizing-the-dsl"></a>Настройка DSL  
 Если вы хотите изменить определение DSL, Закройте экспериментальный экземпляр и обновить определение в главном [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] экземпляра.  
  
> [!NOTE]
>  После изменения определения DSL, вы можете потерять информацию в тестирования моделей, созданных с помощью предыдущих версий.  Например отладки решение содержит файл с именем примера, который содержит несколько фигур и соединителей. После начала разработки Определение DSL, они не будут видны, и они будут потеряны при сохранении файла.  
  
 Можно сделать широкий спектр расширения доменного языка. Следующие примеры обеспечит впечатление возможности.  
  
 После каждого изменения сохранить определения DSL щелкните **преобразовать все шаблоны** в **обозревателе решений**и нажмите клавишу **F5** поэкспериментировать с измененной DSL.  
  
### <a name="rename-the-types-and-tools"></a>Переименование типов и средств  
 Переименование существующих классов доменов и отношений. Например начиная с определения Dsl, созданная из шаблона минимальный язык, может выполнять следующие операции переименования, чтобы DSL представления дерева семейства.  
  
##### <a name="to-rename-domain-classes-relationships-and-tools"></a>Чтобы переименовать классов доменов, связи и средства  
  
1.  В схеме DslDefinition Переименуйте **ExampleModel** для **FamilyTreeModel**, **ExampleElement** для **лица**, **целевые объекты** для **родительских элементов**, и **источников** для **дочерних**. Можно щелкнуть каждый метку, чтобы изменить его.  
  
     ![Схема определения DSL — модель семейного дерева](~/docs/modeling/media/familyt_person.png "FamilyT_Person")  
  
2.  Переименуйте элемент и соединитель средства.  
  
    1.  Откройте окно обозревателя DSL, перейдя на вкладку в обозревателе решений. Если его не видно, на **представление** , выберите в меню **другие окна** и нажмите кнопку **обозреватель DSL**. Обозреватель DSL является видимым только в том случае, когда схема определения DSL — это активное окно.  
  
    2.  Откройте окно свойств и разместить обозреватель DSL и свойств можно увидеть в то же время.  
  
    3.  В обозревателе DSL разверните **редактор**, **вкладки панели элементов**, * \<DSL настроек*, а затем **средства**.  
  
    4.  Щелкните **ExampleElement**. Это элемент панели инструментов, который используется для создания элементов.  
  
    5.  В окне «Свойства» измените **имя** свойства **лицо**.  
  
         Обратите внимание, что **заголовок** также изменяется.  
  
    6.  Аналогичным образом изменить имя **ExampleConnector** средство для **ParentLink**. ALTER **заголовок** свойство, так что он не является копией имени свойства. Например, введите **родительская ссылка**.  
  
3.  Перестройте DSL.  
  
    1.  Сохраните файл определения DSL.  
  
    2.  Щелкните **преобразовать все шаблоны** на панели инструментов обозревателя решений  
  
    3.  Нажмите клавишу F5. Подождите, пока экспериментальный экземпляр [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] отображается.  
  
4.  Отладка решения в экспериментальном экземпляре [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)], откройте файл модели тестирования. Перетащите элементы на его из области элементов. Обратите внимание, что средство заголовки и имена типов в обозревателе DSL изменились.  
  
5.  Сохраните файл модели.  
  
6.  Откройте tt-файл и замените вхождения старые имена типов и свойств новые имена.  
  
7.  Убедитесь в том, что имя файла, который указан в tt-файл указывает модели тестирования.  
  
8.  Сохраните tt-файл. Откройте созданный файл для просмотра результатов выполнения кода в tt-файл. Убедитесь, что он работает.  
  
### <a name="add-domain-properties-to-classes"></a>Добавить свойства доменных классов  
 Добавьте свойства класса домена, например для представления года рождения и смерти человека.  
  
 Чтобы отобразить новые свойства на схеме, необходимо добавить *декораторы* форма, которая отображает элемент модели. Также необходимо сопоставить свойства декораторов.  
  
##### <a name="to-add-properties-and-display-them"></a>Для добавления свойств и их отображения  
  
1.  Добавьте свойства.  
  
    1.  Схема определения DSL, щелкните правой кнопкой мыши **лицо** класс домена, выберите пункт **добавить**и нажмите кнопку **свойства домена**.  
  
    2.  Введите список новые имена свойств, таких как **рождения** и **смерти**. Нажмите клавишу **ввод** после каждого из них.  
  
2.  Добавление декораторов, которое будет отображать свойства в фигуре.  
  
    1.  Выполните серой линии, начиная с класса домена пользователя на другую сторону диаграммы. Это карта элементов схемы. Он связывает класс shape класс домена.  
  
    2.  Этот класс shape щелкните правой кнопкой мыши **добавить**, а затем нажмите кнопку **декоратор**.  
  
    3.  Добавьте два декораторы с именами, например **BirthDecorator** и **DeathDecorator**.  
  
    4.  Выберите каждый новый декоратора, а в окне «Свойства» задайте **позиции** поле. Определяет, где будет отображаться значение свойства домена на фигуре. Например, задать **InnerBottomLeft** и **InnerBottomRight**.  
  
         ![Определение фигуры секции](../modeling/media/familyt_compartment.png "FamilyT_Compartment")  
  
3.  Сопоставление декораторы со свойствами.  
  
    1.  Откройте окно сведений DSL. Обычно это на вкладке рядом с в окне вывода. Если его не видно, на **представление** наведите указатель мыши на **другие окна**, а затем нажмите кнопку **подробные сведения о DSL**.  
  
    2.  В схеме определения DSL щелкните линии, соединяющей **лицо** класс домена для класса shape.  
  
    3.  В **подробные сведения о DSL**на **декораторов** щелкните флажок несопоставленные декоратор. В **отображаемое свойство**, выберите свойство домена, к которому нужно сопоставить. Например, сопоставление **BirthDecorator** для **рождения**.  
  
4.  Сохранить DSL преобразовать все шаблоны и нажмите клавишу F5.  
  
5.  В схеме модели образца убедитесь, теперь щелкните выбранных позиций и введите значения в них. Кроме того, при выборе **лицо** фигуры в окне «Свойства» отображаются свойства нового рождения и смерти.  
  
6.  В tt-файл можно добавить код, который получает свойства каждого пользователя.  
  
 ![Обозреватель, панель элементов и схема семейного дерева](~/docs/modeling/media/familyt_instance.png "FamilyT_Instance")  
  
### <a name="define-new-classes"></a>Определять новые классы  
 Классы доменов и связи можно добавить в модель. Например можно создать новый класс для представления города и создать связь для представления, что человек жили в городе.  
  
 Чтобы сделать distinct различных типов на схеме модели, можно сопоставить классы доменов различные фигуры или фигур с другими параметрами и цвета.  
  
##### <a name="to-add-and-display-a-new-domain-class"></a>Добавить и отобразить новый класс домена  
  
1.  Добавление класса домена и сделайте его дочерним элементом корневого элемента модели.  
  
    1.  В схеме определения DSL щелкните **отношение внедрения** , щелкните корневой класс **FamilyTreeModel**, а затем щелкните пустую область схемы.  
  
         Новый класс домена появится, подключенный FamilyTreeModel с отношение внедрения.  
  
         Присвойте ему имя, например **города**.  
  
        > [!NOTE]
        >  Каждый класс домена кроме корня модели должен быть целевым объектом хотя бы одного отношения внедрения, или он должен наследовать от класса, который является целевым объектом внедрения. По этой причине часто удобно создать класс домена с помощью средства отношение внедрения.  
  
    2.  Добавьте свойство домена для нового класса, например **имя**.  
  
2.  Добавьте отношение ссылки между Person и городе.  
  
    1.  Щелкните **ссылочное отношение** , щелкните пользователя и нажмите кнопку городе.  
  
         ![Фрагмент определения DSL: корень семейного дерева](../modeling/media/familyt_root.png "FamilyT_Root")  
  
        > [!NOTE]
        >  Ссылочные отношения представляют перекрестные ссылки из одной части дерева модели в другую.  
  
3.  Добавление фигуры для города на схемах модели представления.  
  
    1.  Перетащите **геометрическая фигура** из панели элементов на схему и переименуйте его, например **TownShape**.  
  
    2.  В окне «Свойства» задайте внешний вид поля новой фигуры, такие как цвет заливки и Geometry.  
  
    3.  Добавление декораторов для отображения названия города и переименуйте его в NameDecorator. Задайте свойство Position.  
  
4.  Сопоставьте класс домена Город TownShape.  
  
    1.  Щелкните **карту элементов схемы** программу, а затем щелкните класс домена Город и класс shape TownShape.  
  
    2.  В **декораторов** вкладке **подробные сведения о DSL** выбранное окно с соединителем карты, проверьте NameDecorator и установите **отображаемое свойство** имени.  
  
5.  Создайте соединитель для отображения отношения между Person и города.  
  
    1.  Перетащите соединитель из панели элементов на схему. Переименовать его и задайте его свойства внешнего вида.  
  
    2.  Используйте **карту элементов схемы** средство связывание нового соединителя связь между Person и городе.  
  
         ![Определение семейного дерева с добавленной картой фигур](../modeling/media/familyt_shapemap.png "FamilyT_ShapeMap")  
  
6.  Создание средства элемента для внесения городе new.  
  
    1.  В **обозреватель DSL**, разверните **редактор** затем **вкладки панели элементов**.  
  
    2.  Щелкните правой кнопкой мыши * \<DSL настроек* и нажмите кнопку **добавить новое средство элемента**.  
  
    3.  Задайте **имя** свойства нового средства и задать его **класс** свойства к работе.  
  
    4.  Задайте **значок панели элементов** свойство. Click **[...] ** и **имя файла** выберите файл значка.  
  
7.  Создайте соединитель инструмент для создания ссылки между города и людей.  
  
    1.  Щелкните правой кнопкой мыши * \<DSL настроек* и нажмите кнопку **добавить новое средство соединитель**.  
  
    2.  Установите свойство имени нового средства.  
  
    3.  В **ConnectionBuilder** свойства, выберите построитель, который содержит имя человека городе связи.  
  
    4.  Задайте **значок панели элементов**.  
  
8.  Сохранить определения DSL, нажмите кнопку **преобразовать все шаблоны**, а затем нажмите клавишу **F5**.  
  
9. В экспериментальном экземпляре [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)], откройте файл модели тестирования. Используйте новые средства для создания города и связей между города и лиц. Обратите внимание, что можно создать только связи между соответствующими типами элементов.  
  
10. Создайте код, содержащий города, в котором живет каждого пользователя. Текстовые шаблоны являются одним из мест, где выполняется такого кода. Например можно изменить файл Sample.tt отладка решения, чтобы он содержал следующий код:  
  
    ```  
    <#@ template inherits="Microsoft.VisualStudio.TextTemplating.VSHost.ModelingTextTransformation" debug="true" #>  
    <#@ output extension=".txt" #>  
    <#@ FamilyTree processor="FamilyTreeDirectiveProcessor" requires="fileName='Sample.ftree'" #>  
  
    <#  
      foreach (Person person in this.FamilyTreeModel.People)  
      {  
    #>  
        <#= person.Name #><#if (person.Town != null) {#> of <#= person.Town.Name #> <#}#>  
  
    <#  
          foreach (Person child in person.Children)  
      {  
    #>  
                <#= child.Name #>  
    <#  
      }  
      }  
    #>  
  
    ```  
  
     При сохранении файла *.tt, он создаст дочерний файл, содержащий список пользователей и их residences. Дополнительные сведения см. в разделе [создание кода из доменного языка](../modeling/generating-code-from-a-domain-specific-language.md).  
  
## <a name="validation-and-commands"></a>Проверка и команды  
 Можно разработать дальнейшей этот DSL путем добавления ограничения проверки. Эти ограничения являются методы, которые можно определить, убедитесь, что модель находится в правильном состоянии. Например можно определить ограничение, чтобы убедиться в том, Дата рождения дочернего позже родительских элементов. Функция проверки отображает предупреждение, если пользователь DSL пытается сохранить модель, которая разбивает никакие ограничения. Дополнительные сведения см. в разделе [проверки в доменном языке](../modeling/validation-in-a-domain-specific-language.md).  
  
 Можно также определить команды меню, которые пользователь может вызывать. Команды можно изменить модель. Они также могут взаимодействовать с другими моделями в [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] и внешние ресурсы. Дополнительные сведения см. в разделе [Практическое руководство: изменение стандартной команды меню](../modeling/how-to-modify-a-standard-menu-command-in-a-domain-specific-language.md).  
  
## <a name="deploying-the-dsl"></a>Развертывание DSL  
 Чтобы разрешить другим пользователям для использования доменного языка, распространять [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] файл Extension (VSIX). Оно создается при построении решения DSL.  
  
 Найдите VSIX-файл в папке bin вашего решения. Скопируйте его на компьютер, на котором требуется установить его. На этом компьютере дважды щелкните VSIX-файл. DSL может использоваться во всех экземплярах [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] на этом компьютере.  
  
 Можно использовать ту же процедуру для установки DSL на локальном компьютере, не придется использовать экспериментальный экземпляр [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)].  
  
 Дополнительные сведения см. в разделе [развертывание решений доменного языка](../modeling/deploying-domain-specific-language-solutions.md).  
  
##  <a name="a-namereseta-removing-old-experimental-dsls"></a><a name="Reset"></a>Удаление старого экспериментальном DSL  
 Если вы создали экспериментальном DSL, больше не требуется, можно удалить с компьютера, сбросив [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] экспериментальный экземпляр.  
  
 Это удалит с компьютера все экспериментальном DSL и других экспериментальных [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] расширения. Это расширения, которые были выполнены в режиме отладки.  
  
 Эта процедура не удаляет DSL или другой [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] расширения, которые были полностью установлены, выполнив VSIX-файл.  
  
#### <a name="to-reset-the-visual-studio-experimental-instance"></a>Чтобы сбросить Visual Studio экспериментальный экземпляр  
  
1.  Щелкните **запустить**, нажмите кнопку **все программы**, **Microsoft Visual Studio 2010 SDK**, **средства**, а затем **Сброс Microsoft Visual Studio 2010 экспериментальный экземпляр**.  
  
2.  Перестроить все экспериментальном DSL или других экспериментальном [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] расширений, которые хотите использовать.
