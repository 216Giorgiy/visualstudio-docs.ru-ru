# Настройка общих параметров сеанса производительности
[!INCLUDE[vs2017banner](../code-quality/includes/vs2017banner.md)]

Можно задать правила об именовании данных профилирования и методов сбора для сеанса производительности средств профилирования [!INCLUDE[vsprvs](../code-quality/includes/vsprvs_md.md)] на странице **Общие** диалогового окна свойств сеанса производительности.  Чтобы открыть это диалоговое окно из **Обозревателя производительности**, щелкните правой кнопкой мыши сеанс анализа производительности и выберите команду **Свойства**.  
  
 **Требования**  
  
-   [!INCLUDE[vsUltLong](../code-quality/includes/vsultlong_md.md)], [!INCLUDE[vsPreLong](../code-quality/includes/vsprelong_md.md)], [!INCLUDE[vsPro](../code-quality/includes/vspro_md.md)]  
  
## Выбор методов сбора данных  
 Задание основного метода сбора выполняется путем выбора одного из параметров в разделе **Набор профилирования**.  Параметры описаны в следующей таблице.  
  
|||  
|-|-|  
|**Выборка** В методе выборки выполняется сбор сведений профилирования через определенные интервалы.  Этот метод целесообразно использовать для нахождения проблем, относящихся к использованию процессора, а также этот метод предлагается для выполнения большинства операций анализа производительности.|-   [Использование метода выборки для сбора статистики производительности](../profiling/collecting-performance-statistics-by-using-sampling.md)|  
|**Инструментирование**.  В методе инструментирования выполняется вставка кода профилирования в копию модуля, который записывает каждое событие входа, выхода, вызова функции для функций модуля в сеансе профилирования.  Этот метод целесообразно использовать с целью сбора подробных сведений о разделе кода для понимания влияния операций ввода\-вывода на производительность приложения.|-   [Сбор подробных сведений о времени с помощью инструментирования](../profiling/collecting-detailed-timing-data-by-using-instrumentation.md)|  
|**Параллелизм**.  В методе параллелизма выполняется сбор данных для каждого события, которое блокирует выполнение кода, например, когда поток ожидает освобождения доступа к заблокированному ресурсу приложения.  Этот метод целесообразно использовать при анализе многопоточных приложений.|-   [Сбор данных о параллелизме потоков и процессов](../profiling/collecting-thread-and-process-concurrency-data.md)|  
  
 Данные памяти .NET можно собрать с использованием методов выборки или инструментирования.  Тип данных можно выбрать в области **Набор профилирования памяти .NET**.  
  
|||  
|-|-|  
|**Собирать сведения о выделении объектов .NET**.  По умолчанию данные содержат сведения о количестве и размерах выделенных объектов.  Установите или снимите этот флажок, чтобы включить или выключить сбор сведений о выделении объектов .NET.<br /><br /> **Также собирать сведения о времени жизни объектов .NET**.  Установите этот флажок, чтобы включить данные о поколениях сборов мусора, которые используются для удаления объектов памяти.|-   [Сбор данных о выделении памяти для объектов .NET и времени их жизни](../profiling/collecting-dotnet-memory-allocation-and-lifetime-data.md)|  
  
 Страница сеанса профилирования отобразится при запуске профилирование приложения, на ней можно приостановить, возобновить и остановить профилирование.

## Настройка параметров файла данных профилирования  
  
|||  
|-|-|  
|**Отчет**.  По умолчанию файлу данных профилирования \(.vsp\) дается имя профилируемого приложения, и он располагается в папке проекта или решения.  К именам файлов данных также добавляется строка с датой и номер с приращением для предупреждения создания одинаковых имен.  Эти параметры можно изменить.|-   [Практическое руководство. Настройка параметров имени файла данных профилирования](../profiling/how-to-set-performance-data-file-name-options.md)|
