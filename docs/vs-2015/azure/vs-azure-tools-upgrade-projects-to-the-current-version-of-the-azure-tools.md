# <a name="how-to-upgrade-projects-to-the-current-version-of-the-azure-tools-for-visual-studio"></a>Обновление проектов до текущей версии инструментов Azure для Visual Studio
## <a name="overview"></a>Обзор
После установки текущего выпуска инструментов Azure (или предыдущего выпуска, который новее, чем 1.6), все проекты, созданные с помощью инструментов Azure выпущенных до версии 1.6 (ноябрь 2011 г.) будут автоматически обновлены сразу же после их открытия. Если вы создавали проекты с помощью этих средств выпуске 1.6 (ноябрь 2011 г.), и у вас по-прежнему есть что установлен выпуск, можно открывать такие проекты в старой версии и было позднее принять решение следует ли их обновлять.

## <a name="how-your-project-changes-when-you-upgrade-it"></a>Изменение проекта при его обновлении
Если проект обновляется автоматически, или указать, что вы хотите обновить ее, проект будет модифицирован для работы с текущими версиями определенных сборок, и как в этом разделе описываются некоторые свойства также можно изменить. Если проект требует других изменений для обеспечения совместимости с новой версией средства, эти изменения необходимо вносить вручную.

* Файл web.config для веб-ролей и файл app.config для рабочих ролей обновляются для ссылки на более новой версии Microsoft.WindowsAzure.Diagnostics.DiagnosticMonitoirTraceListener.dll.
* Сборки Microsoft.WindowsAzure.StorageClient.dll, Microsoft.WindowsAzure.Diagnostics.dll и Microsoft.WindowsAzure.ServiceRuntime.dll обновляются до новых версий.
* Профили публикации, которые были сохранены в файле проекта Azure (.ccproj), перемещаются в отдельный файл с расширением azurepubxml в **публикации** подкаталог.
* Некоторые свойства в профиле публикации обновляются для поддержки новых и измененных функций. **AllowUpgrade** заменяется **DeploymentReplacementMethod** поскольку развернутую облачную службу можно обновлять одновременным или добавочным способом.
* Свойство **UseIISExpressByDefault** добавляется и значение false, чтобы веб-сервер, который используется для отладки не изменялся автоматически с Internet Information Services (IIS) на IIS Express. IIS Express является сервером по умолчанию для проектов, созданных с помощью новых версий инструментов.
* Если Azure Caching размещается в одном или нескольких ролях вашего проекта, при обновлении проекта некоторые свойства в конфигурации службы (cscfg-файл) и определения службы (csdef-файле) изменяются. Если проект использует пакет NuGet для кэша Azure, проект обновляется до последней версии пакета. Следует открыть файл web.config и убедитесь, что конфигурация клиента поддерживалась должным образом во время обновления. Если вы добавили ссылки на клиентские сборки Azure Caching без использования пакета NuGet, эти сборки не будут обновлены; необходимо вручную обновить эти ссылки до новых версий.

> [!IMPORTANT]
> Для F# проектов, необходимо вручную обновить ссылки на сборки Azure, чтобы они ссылались на более новые версии этих сборок.

> 

### <a name="how-to-upgrade-an-azure-project-to-the-current-release"></a>Обновление проекта Azure до текущего выпуска
1. Установить текущую версию инструментов Azure в процессе установки Visual Studio, который вы хотите использовать для обновленного проекта, а затем откройте проект, который требуется обновить. Если проект был создан с помощью средств Azure выпущенных до версии 1.6 (ноябрь 2011 г.), он автоматически обновляется до текущей версии. Если проект был создан с ноябрь 2011 г. выпуска и этот выпуск все еще установлен, проект откроется в этом выпуске.
2. В обозревателе решений откройте контекстное меню для узла проекта, выберите **свойства**, а затем выберите **приложения** диалоговое окно, которое отображается на вкладке.
   
    **Приложения** вкладке отображается версия инструментов, который связан с проектом. Если отображается текущая версия инструментов Azure, проект уже обновлен. Если вы установили более новую версию средства, чем показано на этой вкладке, **обновление** появится кнопка.
3. Выберите **обновление** кнопку, чтобы обновить проект до текущей версии инструментов.
4. Постройте проект и затем исправьте все ошибки, возникающие в результате изменения API. Сведения о том, как изменить код для новой версии см. в документации к конкретному API.
