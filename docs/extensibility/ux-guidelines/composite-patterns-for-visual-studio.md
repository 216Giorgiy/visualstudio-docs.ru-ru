##  <a name="a-namebkmkdatavisualizationa-data-visualization"></a><a name="BKMK_DataVisualization"></a> Визуализация данных  
  
### <a name="overview"></a>Обзор  
 Диаграммы являются наглядный способ объединения и визуализации данных, чтобы улучшить процесс принятия решений. Они могут помочь пользователи сталкиваются с большим количеством данных, но немного это означает, что заслуживает внимания и то, что может потребоваться действие.  
  
 Пользователь будет обеспечен диаграммы при выполнении любого из следующих условий:  
  
-   Диаграмма поможет пользователям определить задачи, которые они могут выполнять?  
  
-   Диаграммы позволяют пользователям прогноз последствия потенциальных изменений?  
  
-   Диаграммы помогут пользователям трендов и выявить закономерности?  
  
-   Диаграмма позволит пользователям принимать более обоснованные решения?  
  
-   Диаграмма поможет ответить на конкретный вопрос, возможно, пользователи в данном контексте?  
  
#### <a name="general-rules-for-charts"></a>Общие правила для диаграмм  
  
-   Явно пометить данные. Иллюстрации без объяснения не так просто просто картинки.  
  
-   Запустите осей с нуля, чтобы избежать наклона пропорции. Размер строки длины и строки, важные визуальные подсказки для понимания связей между точками данных.  
  
-   Создание диаграмм, не инфографика. Инфографика, художественных представления данных и их основной целью является visual повествования. Диаграммы можно (и нужно быть визуально привлекательные, но позвольте данным говорят сами за себя.  
  
-   Избегайте skeumorphism, наглядные гистограмм, hashmarks контрастность и других прикосновений среда.  
  
-   Не используйте объемные эффекты как декоративный элемент. Использовать их только в том случае, если они действительно неотъемлемой частью возможность воспринимать информацию.  
  
-   Избегайте несколько линий и заливок, как более двух цветов может затруднить этот тип диаграммы для чтения и правильно интерпретировать.  
  
-   Не используйте диаграмму (или любой рисунке) единственным средством понимание концепции или взаимодействия с данными. Это представляет трудности для пользователей с нарушениями зрения.  
  
-   Не используйте диаграммы как предварительный или декоративных элементов на странице. Другими словами Если диаграммы не добавить всех пользователей значение или справки в решении проблемы, не используйте его.  
  
### <a name="chart-types"></a>Типы диаграмм  
 Типы диаграмм, используемых в Visual Studio включают гистограммы, графики, измененные круговой диаграммы, известный как кольцевой диаграммы или «кольцо диаграмма», временные шкалы, Точечная диаграмма (также называется «диаграммы кластера») и диаграммы Ганта. Каждый тип диаграмм полезно для взаимодействия свой тип сведений.

##  <a name="a-namebkmkonobjectuia-on-object-ui-and-peeking"></a><a name="BKMK_OnObjectUI"></a> На объект пользовательского Интерфейса и просмотр  
 В этом разделе приводится контекст на просмотр, также известный как просмотр в представлении кода, тип объекта пользовательского Интерфейса, уникальные для Visual Studio.  
  
### <a name="overview"></a>Обзор  
  
-   Пользовательского Интерфейса на объект должен предоставить пользователю Дополнительные сведения или интерактивность без отвлекать от их основных задач.  
  
-   Основной шаблон для объектов пользовательского Интерфейса в Visual Studio называют «сведения о точке внимания».  
  
-   На объект пользовательского Интерфейса в Visual Studio — встроенный или с плавающей запятой и либо устойчивыми или временной.  
  
    -   Просмотр в представлении кода, тип объектов пользовательского Интерфейса в Visual Studio является встроенной и длительного пользования.  
  
    -   CodeLens, тип объектов пользовательского Интерфейса в Visual Studio с плавающей запятой и временных  
  
 Поняв, как работает фрагмент кода или поиск сведений об этом коде часто требует разработчиков для переключения контекста и перейти на другое содержимое или другой окна. Эти смены контекста могут оказаться разрушительными, поскольку пользователи могут потерять сосредоточиться на своей исходной задачи, если они покидают их главное окно. Кроме того получение исходного контекста обратно, что может быть сложно, особенно в том случае, если переключение окон за их исходный код быть скрыт другой пользовательский Интерфейс.  
  
 На объект пользовательского Интерфейса соответствует шаблону, называется «сведения о точке внимания». Предоставить пользователям дополнительные соответствующие сведения, добавляющий разъяснение или интерактивность без потери сосредоточиться на своих основных задач, эти сообщения, всплывающие окна и диалоговые окна. Примеры объектов пользовательского Интерфейса: всплывающие окна, которые появляются при наведении указателя их на значок в области уведомлений, красной волнистой линией неправильно написанному слову и считывания представления, представленные в Visual Studio 2013.  
  
### <a name="decision-points"></a>Точки принятия решений  
 В Visual Studio Чтобы использовать этот шаблон информации на момент внимания несколькими способами. Выбор механизма вправо и ее реализации в виде единообразного, предсказуемого важно общее удобство работы. В противном случае — пользователи могут осуществлять затрудняет фокуса из содержимого самого работу вводят в заблуждение или не согласована.  
  
#### <a name="relationships-between-master-and-detail-content"></a>Связи между основными и подробными содержимого  
 Сведения о точке внимания используется для отображения отношения между содержимым, является ли пользователь конкретное («основной» содержимого) и дополнительные связанные content (содержимое «подробности»). В этом шаблоне детализации содержимого тесно связан с содержимым пользователь работает с и могут быть отображены рядом содержимого главной. Дополнительные сведения или сведения, которые не могут быть сведены в не забрасывая главного содержимого необходимо следовать другой шаблон, например окно инструментов.  
  
-   **Всегда** отображения содержимого детализации в непосредственной близости от главного содержимого.  
  
-   **Всегда** Убедитесь, что содержимое сведений по-прежнему дает пользователю возможность уделять больше внимания содержимого главной. Часто лучший способ добиться этого — визуализации содержимого подробно как близко к содержимого главной максимально. Это можно сделать визуализации содержимого сведений во всплывающем окне рядом с содержимым, master или визуализации встроенное содержимое детализации под главной содержимого.  
  
-   **Никогда не** использование информации на момент перехода пользователя от содержимого главной внимания. Если пользователям требуется просмотреть содержимое подробно отдельно, предоставлять явное действие, которое позволяет пользователю это сделать.  
  
#### <a name="design-details"></a>Подробные сведения о макете  
 После определения того, что пользовательского Интерфейса на объект является правильным выбором, существует четыре основных моментах:  
  
1.  **Сохранение:** содержимое должно быть постоянными или временными?   
    Будет необходимо закрепить данные называют или взаимодействие с? Или будет пользователям быстро взглянуть на данные, а затем продолжите их основные задачи?  
  
2.  **Тип содержимого:** содержимое будет информационное, готовые к работе или переходов?   
    Пользователю нужны дополнительные сведения о содержимом, master? Необходимо выполнить задачу, которая влияет на содержимое главной пользователя? Или пользователь должен направляться на другой ресурс?  
  
3.  **Тип индикатора:** окружения индикатора смысл?   
    Сведения целом удобным способом, а не отображается, не забрасывая содержимого главной страницы?  
  
4.  **Жесты:** жесты, что будет использоваться для вызова неуправляемого кода и закрыть пользовательский Интерфейс?   
    Как пользователь открыть содержимое детализации и отправить его сейчас? Существует значение, при добавлении жеста, такие как закрепление для переключения между состояниями временные и устойчивые?  
  
 Каждая из этих четырех решению повлияет на основные компоненты пользовательского Интерфейса объекта.  
  
### <a name="on-object-ui-components"></a>Компоненты пользовательского Интерфейса на объект  
  
1.  Тип контейнера (презентатор контента)  
  
    -   Числа с плавающей запятой  
  
    -   Встроенная функция  
  
2.  Тип содержимого  
  
    -   Информация: данные, которые могут быть статическими или динамическими  
  
    -   Готовые к работе: команд, позволяющих изменить содержимого главной страницы  
  
    -   Навигации: ссылки ведут к другому окну или приложения, например MSDN  
  
3.  Жесты  
  
    -   Вызов  
  
    -   Увольнения  
  
    -   Закрепление  
  
    -   Другие случаи взаимодействия  
  
4.  Сохранение состояния и фиксация модели  
  
    -   Временные  
  
    -   Durable  
  
    -   Автоматический  
  
    -   По запросу  
  
5.  Внешняя индикаторы (необязательно)  
  
    -   Подчеркивание волнистой линией  
  
    -   Значок смарт-тега  
  
    -   Другие показатели окружающей среды  
  
#### <a name="container-content-presenter-type"></a>Тип контейнера (презентатор контента)  
 Существует два основных варианта для представления содержимого точке внимания:  
  
1.  **Встроенный:** встроенный презентатора, например представление считывания, которая появилась в редакторе кода Visual Studio 2013 освобождая место для нового содержимого сдвиг существующего содержимого.  
  
    -   **Предпочтение** представить встроенный докладчиков, если ожидается, что пользователям нужно провести немало времени, ссылающегося на или взаимодействие с содержимым.  
  
    -   **Избегайте** встроенный докладчиков, если ожидается, что пользователи хотите взглянуть на данные можно представить, а затем продолжить их основные задачи с минимальными простоями.  
  
2.  **С плавающей запятой:** плавающей презентатор расположен близко к выбранное содержимое можно, но не изменяет макет существующего содержимого. Можно применять различные стратегии, например при отображении плавающей панели содержимого через ближайших доступных пробелы к выделенному символу.  
  
    -   **Предпочтение** плавающей докладчиков, если ожидается, что пользователи будут посмотрите на информацию, можно представить, а затем продолжить их основные задачи с минимальными простоями.  
  
    -   **Избегайте** плавающей докладчиков, если ожидается, что пользователи будут нужно тратить немало времени, ссылающегося на или взаимодействие с содержимым при наличии.  
  
#### <a name="content-type"></a>Тип содержимого  
 Существует три основных типа содержимого, могут отображаться в любом контейнере пользовательского Интерфейса на объект. Любое сочетание этих типов данных могут быть отображены. Ниже приведены три типы.  
  
1.  **Информация:** большинство объектов контейнеров пользовательского Интерфейса будет отображаться информационное содержимое любого рода. Содержимое может представлять сведения о текущем состоянии среды или он может представлять сведения о потенциальных будущего состояния среды. Например он может использоваться, чтобы показать эффект определенной команды, например рефакторинг, в существующий код.  
  
    -   **Всегда** использовать каноническое представление данных, которые можно отобразить. Например код должен выглядеть код, включающий подсветку синтаксиса и должен подчиняться шрифта и другие параметры среды, заданные пользователем.  
  
    -   **Всегда** рассмотреть возможность поддержки каких-либо действий над информационное содержимое, которое бы Если этой информации представляется в виде содержимого главной страницы. Например если представление существующий код внутри пользовательского Интерфейса на объект контейнера, настоятельно рекомендуется поддержка возможности просмотра и изменения этого кода.  
  
    -   **Всегда** рекомендуется использовать другой цвет фона при предоставлении информационное содержимое, которое представляет потенциальных будущего состояния.  
  
2.  Готовые к работе: некоторые контейнеры пользовательского Интерфейса на объект предоставит возможность выполнения некоторых операций над главного содержимого, например операции рефакторинга.  
  
    -   **Всегда** положение принятия команд отдельно от информационное содержимое.  
  
    -   **Всегда** включать и отключать действия при необходимости.  
  
    -   **Всегда** ссылаться на стандартные правила для представления команд в диалоговых окнах.  
  
    -   **Всегда** Сохранить минимальное количество действий, представленных в контейнер пользовательского Интерфейса на объект абсолютное значение. Взаимодействие с пользовательским Интерфейсом объекта должно быть упрощенный, быстрый качества. Пользователю не следует тратить энергию на управление сам контейнер пользовательского Интерфейса на объект.  
  
    -   **Всегда** рассмотрим, как и когда контейнер пользовательского Интерфейса на объект будет закрыт или отброшено. По соображениям любое действие, которое завершает диалог между основными и подробными содержимого следует закрыть контейнер пользовательского Интерфейса на объект при вызове этого действия.  
  
3.  **Навигационные:** некоторых объектов пользовательского Интерфейса контейнеры содержат ссылки, которые ведут к другому окну или приложения, например открытие статьи MSDN в веб-браузере пользователя.  
  
    -   **Всегда** Начало все навигационные ссылки с «Открыть», чтобы пользователи будут получать уведомления, куда выполняется переход на некоторое другое содержимое.  
  
    -   **Всегда** отделения ссылок для перехода от принятия ссылки.  
  
#### <a name="ambient-indicators-optional"></a>Внешняя индикаторы (необязательно)  
 Окружения индикаторы могут быть небольшие, включая текст, представленный в контрастных цветов от остальной части кода, или очевидно, включая tickler символы, такие как подчеркивание волнистой линией и значки смарт-тег. Индикаторы внешней информацией о доступности соответствующих дополнительных сведений. В идеале они окажутся полезными даже без взаимодействия с пользователем, для взаимодействия с ними.  
  
-   **Всегда** положение индикатора окружающей среды, чтобы не сбивать с толку или перегрузки пользователя. Если это невозможно, для размещения внешнего индикатора таким образом, рекомендуется другое решение.  
  
-   **Всегда** максимально близко к содержимому, которое относится к позиции внешнего индикатора.  
  
-   **Всегда** Попробуйте создать индикатор, обобщающий сведения, это делает доступными. Следует указать число данных элементов (например, «3 ссылки» вместо просто «ссылки») или представить некоторые другие способы объединения данных.  
  
    -   В случаях, когда данные для индикатора нельзя всегда вычисляется и отображается немедленно следует рассмотреть возможность создания последовательной отзыв как вычисляются значения. Например рассмотрим анимация изменения, отражающие обновлений, доступные данные аналогичны тем, что обновляет динамическую плитку электронной почты на Windows Phone как число увеличивается непрочтенных сообщений электронной почты.  
  
-   **Никогда не** Добавить дополнительные индикаторы, чем пользователь может выполнять достаточно для данной части содержимого. Индикаторы окружения следует полезно без необходимости вмешательства пользователя. Индикаторы утрате их окружением они требуют переполнения и другие средства управления для их отображения.  
  
#### <a name="gestures"></a>Жесты  
 Ключевым аспектом разрешить пользователю Концентрируйтесь на содержимого главной — благодаря поддержке справа жесты для открытия и закрытия содержимое дополнительных сведений.  
  
-   **Всегда** пользователю необходимо выполнить некоторые явный жест, чтобы открыть дополнительное содержимое. Общие открытые жесты включают:  
  
    -   **При наведении курсора мыши:** подсказки или неинтерактивном информационное содержимое  
  
    -   **Команда явно:** встроенный докладчика  
  
    -   **Дважды щелкните индикатор окружения:** CodeLens всплывающего окна  
  
-   **Всегда** Закрыть детализации содержимого при каждом нажатии клавиши Esc.  
  
-   **Всегда** учитывать контекст пользовательского интерфейса на объект. Для содержимого докладчиков, обеспечивающие взаимодействие внутри контейнера внимательно рассмотрите возможность Показать дополнительные сведения при наведении курсора мыши, который вероятнее нарушает работу пользователя рабочего процесса.  
  
-   **Никогда не** отображения содержимого при наведении курсора мыши, воспринимается как редактируемый или приглашает взаимодействия с пользователем. Это поведение может разочаровать пользователей, если они пытаются курсор на содержимое подробно, как стандартное поведение для всплывающей подсказки немедленно прекратить, когда указатель мыши выходит за пределы сводного содержимого, который его создал.  
  
##  <a name="a-namebkmkselectionmodelsa-selection-models"></a><a name="BKMK_SelectionModels"></a> Выбор модели  
  
### <a name="overview"></a>Обзор  
 Модель выделения — это механизм, используемый для указания и подтверждения операций на один или несколько объектов, имеющих пользовательский интерфейс. В этом разделе рассматриваются варианты взаимодействия выделения в редакторе Visual Studio документ: текстовых редакторов, поверхности разработки и моделирования поверхности.  
  
 Пользователи должны иметь способ показать Visual Studio, что они работают, и Visual Studio должен вернуть предсказуемое отзывы пользователей о то, что он работает. Различия или неправильным обменом данными между пользователем и пользовательский интерфейс может привести к пользователю, не замечают действием, которое может иметь непредвиденные последствия. Часто ошибки остается незамеченной, пока пользователь увидит нечто отсутствует или был изменен. Выбор моделей, следовательно, являются одним из наиболее важных элементов пользовательского интерфейса. Выбор моделей в Visual Studio соответствуют Windows, имеют незначительные изменения.  
  
 В Visual Studio, как и в Windows Выбор модели различаются в зависимости от контекста, в котором происходит взаимодействие. Выбор может осуществляться четыре типа объектов.  
  
-   Text  
  
-   Графические объекты  
  
-   Списки и деревья  
  
-   Таблицы  
  
 В этих объектах существует три типа выделения:  
  
-   Непрерывный  
  
-   Раздельные  
  
-   Region  
  
#### <a name="scope"></a>Область  
 Наиболее важным компонентом выбора является обеспечение пользователь знал, в котором окне они работают (активация) и, задачей которого является найти (Выбор). Visual Studio расширяет возможности управления окна в Windows, но схема активации одинаково: взаимодействие с окном перемещение фокуса в окне. Visual Studio имеется два индикатора для активации: один для окна документа и один для окон инструментов.  
  
 Для окна документа активного окна указывается вкладки окна документа, поступающие на передний план и изменив его цвет фона:

#### <a name="context"></a>Контекст  
 Visual Studio была предназначена для сохранения надежный концепцию контекста, отслеживание которых пользователь работает. Только одно окно активен, является ли окно инструментов или документа. Однако окно переднего плана всегда сохраняет скрытые выделения. Несмотря на то, что в окне инструментов может быть фокус, окна документа, которая была активна отображает, даже в неактивном состоянии. Это позволяет сохранить контекст пользователя в документе, в которую они изменения, показывающий, что Visual Studio сохранил свое состояние, чтобы они могут возвращать и легко перемещение между окнами документов и окнами.  
  
### <a name="text-selection"></a>Выделение текста  
 Visual Studio редакторах строго текстовое, таких как встроенные текстовый редактор, используют ту же модель выделения текста и внешний вид описаны в [и указатели мыши](https://msdn.microsoft.com/en-us/library/dn742466.aspx) страницу Windows взаимодействия руководство на сайте MSDN. Фокус ввода в текстовом редакторе обозначается символом вертикальной черты, именем курсора. Точка вставки находится один пиксель толстые и цветных как обратное все, что находится за его. Мигает, согласно задается скорость **скорость мерцания** в **скорость** Вкладка **клавиатуры** на панели управления.  
  
#### <a name="contiguous-and-disjoint-selection"></a>Выбор смежных и несвязанным  
 Выбор текстового редактора только является непрерывным. Раздельные текстовые выбранные параметры не допускаются, но должно быть устранено в редакторах графический объект. При указателя мыши на область текста, курсор примет форму. Одним щелчком мыши курсор в текстовом редакторе щелкните на месте. Удерживая нажатой кнопку мыши начинается выделение и отпустить кнопку мыши заканчивается выделение.  
  
#### <a name="region-selection-box-selection"></a>Выбор региона (выделение)  
 Visual Studio поддерживает выбор региона в текстовом редакторе, и это называется выделения. Поле выбора позволяет пользователю выбрать область текста, не соблюдаются поток обычный текст. Как и в случае выбора стандартного текста, выбор должны быть смежными. Выделение области инициируется, удерживая нажатой клавишу Alt при перетаскивании мышью. Также можно инициировать выделения, удерживая клавишу Alt и клавиши Shift с помощью клавиш со стрелками для указания области выделения. Выделение области использует обычный выделение и показывает, мигает в конце области выбора курсора точки вставки.

### <a name="graphical-selection"></a>Графическое выделение  
  
#### <a name="interaction"></a>Взаимодействие  
 Выбор графических объектов может быть сложным и зависит от ряда факторов:  
  
-   **Основной выделенный объект модели редактора.** Редакторы, которые содержат графические объекты могут также использоваться для редактирования текста или таблицы. Например редактор может быть текстовый редактор также поддерживает размещение графических объектов, таких как конструктор Visual Studio XAML. Поддержка нескольких типов объектов может повлиять на том, как пользователь выбирает группы состоят из различных типов объектов.  
  
-   **Поддержка состояния основного и дополнительного выбора.** Редактор предоставляет выбор основного и дополнительного состояния, чтобы объекты можно редактировать в конфигурациях, выровнены друг относительно друга, размеры вместе, и т. д.  
  
-   **Поддержка редактирования на месте.** Редакторы можно также разрешить содержимое их графических объектов, который нужно изменить. Например прямоугольник также может содержать текст внутри, которые могут быть изменены пользователем. Кроме того текст может быть по центру или по ширине. Редактирование на месте включает в себя более подробный уровень взаимодействия с пользователем и требует соответствующего набора визуальные подсказки для представления сведений о состоянии для пользователя.

Если редактор предоставляет контейнеры для других графических объектов, приведенные ниже выбора модели должны использоваться только контейнера (объекты внутри контейнера дальнейшие стандартную модель, как описано выше):

### <a name="view-selection-models"></a>Выбор модели представлений  
  
#### <a name="tree-view"></a>Иерархическое представление  
 Выбор в виде дерева показан простой цветом. Если пользователь щелкает имя узла или значок узла, узел выделяется. Треугольное глифы слева от узла разверните или контракта дерево, но не влияют на выбор пользователя, за одним исключением: после свертывания родительский узел, если выделен дочерний этого узла, выделение перемещается к родительскому элементу.  
  
 ![Типичное представление в виде дерева в Visual Studio](../../extensibility/ux-guidelines/media/0713-25_treeview.png "0713-25_TreeView")  
  
 **Типичный дерева в Visual Studio**  
  
 В древовидном представлении может поддерживать выбранные смежных и несвязанным, даже на нескольких уровнях дерева. Смежных или несвязанным множественный выбор должны вноситься на узлы дерева видимым. При сворачивании узла несвязанным выделение теряется, и узел, который был свернут получает выделение. Таким образом пользователь может видеть узлы, которые будут затронуты операцией. Свернуть узлы, становится непонятный какие узлы могут влиять.  
  
 Если выбран родительский узел, следует применить операцию для родительского элемента, хотя возможны случаи, когда имеет смысл операции применяются к родительской и всех его дочерних элементов. В этом случае предоставить дополнительный пользовательский Интерфейс во время операции, например, флажок или диалоговое окно подтверждения, чтобы сделать явную параметр «применяются ко всем дочерним» для пользователя.  
  
##### <a name="renaming"></a>Переименование  
 Если узлы в дереве поддерживает переименование, переименование должно выполняться на месте. Операции с заменой должны быть стандартный для всех элементов управления "дерево" в Visual Studio. Предоставить команде rename, тут же активирует режим редактирования на месте с выделенного текста, охватывающие весь имя узла, готов принимать входные данные пользователя. Если узел представляет файл, имя файла должен содержать расширения. Выделение должно включать имя файла и расширение не только тело.  
  
|Ввод|Результат|  
|-----------|------------|  
|ВВОД - клавиша|Фиксирует операции переименования|  
|Клавиша ESC|Отменяет операцию переименования|  
|Щелкните вне области редактирования на месте|Фиксирует операции переименования|  
|Отменить|Обеспечить возможность легко отмены для отмены операции переименования|  
  
#### <a name="selection-within-lists-and-grid-controls"></a>Выбор в списки и элементы управления grid  
 Ключевая концепция в списке выбора — что на основе строк, это означает, что после выбора всей строки будет выбрана как единое целое. Напротив сетки можно разрешить определенные ячейки выделяется без влияния на другие аспекты строки. Таблицы могут также содержать иерархию вложенных строк (например, представления в виде дерева), позволяющие все ветви иерархии выделить и выбран взаимодействие родительских строк. Выбор в списках показано простое цветом на всю строку данных. Фокус показано пиксель пунктирная граница вокруг текущей редактируемой строки или ячейки (строки, если все ячейки доступны только для чтения).  
  
> [!NOTE]
>  **Фокус** и **выбора** представляют собой разные понятия. *Фокус* указывает на то, какие пользовательского интерфейса целевой элемент для получения входных данных, явно не направлены на другой объект, а *выбора* относится к состоянию объекта включения в набор объектов, на которых последующие операции могут вступить в силу.  
  
 Выбранные элементы в списки могут быть непрерывными, несвязанным, или регион. Когда множественный выбор разрешен, смежных и несвязанным выбора следует всегда будет поддерживаться, во время поддержка Выбор региона (поле) является необязательным. Выбор региона инициируются путем перетаскивания пробелами список текста.  
  
|Объект|Выбор|  
|------------|---------------|  
|Список|Непрерывный|Всегда поддерживается (если множественный выбор разрешен).|  
|Список|Раздельные|Всегда поддерживается (если множественный выбор разрешен).|  
|Список|Region|Дополнительная поддержка. Инициировать, перетаскивая указатель мыши в пустом месте списка текста.|  
  
 Один раз щелкнуть список выделяет строку, где был выполнен щелчок. Если пользователь щелкнет щелкните ячейку списка, который поддерживает изменение на месте, ячейки также немедленно активируется для редактирования на месте. В противном случае — строку целиком немедленно выбран и отображается выделения.  
  
 Перетаскивание в теле список выполняет одно из трех действий:  
  
-   Инициирует Выбор региона, если список поддерживает его и кнопок мыши пустое пространство  
  
-   Инициирует операцию перетаскивания, если список ячейку или строку поддерживает будет источником перетаскивания  
  
-   Выделение текущей строки  
  
##### <a name="in-place-editing"></a>Редактирование на месте  
 Когда разрешено редактирование на месте, существуют две основные модели: Выбор простого редактирования элемента управления и свойства. С простой элемент управления содержимое является выделенным и готов для пользовательского ввода по мере редактирования на месте активируется. При реализации Выбор свойства кнопки, который вызывает выбор свойства отображается после активации режима редактирования на месте, и текущего выделения не выделяется. Кнопка выбора должна быть — по правому краю ячейки. Примеры редактирования на месте см **окно свойств** и **список задач** в Visual Studio.  
  
##### <a name="keyboard-support"></a>Поддержка клавиатуры  
 Поддержка клавиатуры для выбора в списки и таблицы следует стандартным соглашениям Windows:  
  
-   Клавиши со стрелками перехода по списку, выбрав каждой строки или ячейки, при перемещении фокуса.  
  
-   SHIFT + СТРЕЛКА выполняет смежные в направлении клавиш со стрелками.  
  
-   CTRL + стрелка, следуют пробел переключение между Добавление и удаление элементов списка из выделенного фрагмента, Создание несвязанного выделения.  
  
-   Для таблицы, которые содержат вложенные иерархии СТРЕЛКА ВПРАВО разворачивает родительской строки, и клавишу со стрелкой влево сворачивает один.  
  
-   Фокус перемещается клавишей Tab между ячейками в текущей строке ячеек, для редактирования.  
  
-   Клавиши ВВОД выполняет команды по умолчанию для элемента в списке (часто **Откройте**).  
  
-   Клавишу F2 активация на месте редактирования для текущей выбранной ячейки.

### <a name="window-configurations"></a>Конфигурации окна  
 Конфигурации окна является основные представления среды разработки — это схема, состоящий из списка окон инструментов присутствует и упорядочения. Для windows, управляемых в интегрированной среде разработки (IDE windows) сведения о макете сохраняется на пользователя, когда пользователь запускает, интегрированная среда Разработки макета окна появится таким же, как действуют при выходе из Visual Studio. Состояние и положение окон интегрированной среды Разработки, сохраняются в файле пользовательских параметров в формате XML. Окна инструментов, созданные пакеты, загруженный в IDE сохранения информации о состоянии в реестре и может быть не каждого пользователя.  
  
#### <a name="profile-specific-layouts"></a>Макеты определенного профиля  
 Каждый профиль включает средство окон, организованных способом, знакомым персонажи конкретного разработчика (Visual C++ разработчики ожидать **обозревателе решений** левой части интегрированной среды Разработки, а разработчики C# ожидать **обозревателе решений** справа). Макеты окон определенного профиля загружаются после выбора профиля при запуске. Автор пакета следует определить макет окна, наилучшим образом подходят для их улучшения качества, зная, что изменения, внесенные пользователем в конфигурации окна будет сохраняться.  
  
##  <a name="a-namebkmktouchinputa-touch-input"></a><a name="BKMK_TouchInput"></a> Сенсорный ввод  
 Пользователи все чаще используют разработке продуктов Microsoft на сенсорных устройств. Однако существуют препятствий, усложняющих использовать средства разработки на сенсорных устройств. Пользователи ожидают от наших продуктов для обеспечения touch надежным и точным. Цель данного руководства — принять решение о возможностями касания, внедрять и поощрять согласованного сенсорного между Visual Studio и связанных продуктов.  
  
### <a name="levels-of-experience"></a>Уровни качества  
 Следующие уровни качества предназначены для использования в качестве руководства помогают командам решить какие сенсорный предложить на основе их нужного уровня инвестиций интересующим touch.  
  
-    **Базовые функции** для групп, которым требуется предоставить touch возможности, поэтому не тупик во всей своей работы.  
  
-    **Оптимизированные возможности** предназначен для групп, которым требуется предоставить наиболее распространенные сенсорный (например, которые обычно доступны в Интернет-приложения браузера).  
  
-    **Повышением качества** для команд, которые требуется добавить такие возможности как жесты или другие дополнительные возможности, которые могут сделать свои приложения сенсорной понятное.

#### <a name="gestures"></a>Жесты  
 Жесты предоставляют пользователям ярлык для команд, которые в противном случае может потребоваться более сложным взаимодействием. См. в Windows рекомендации по [общего сенсорный ввод для классических приложений](http://msdn.microsoft.com/en-us/library/windows/desktop/dd940543\(v=vs.85\).aspx), и следовать приведенным ниже инструкциям для большинства жестов, включая простых жестов, например панорамирования и масштабирования.
