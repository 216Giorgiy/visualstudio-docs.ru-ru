##  <a name="BKMK_DataVisualization"></a> Визуализация данных  
  
### <a name="overview"></a>Обзор  
 Диаграммы являются визуальный способ статистической обработки и отображения данных, чтобы улучшить процесс принятия решений. Они могут помочь пользователи сталкиваются с большим количеством данных, но практически это означает, что заслуживает внимания и то, что может потребоваться действие.  
  
 Пользователь смогут воспользоваться преимуществами диаграммы, если хотя бы одно из следующих условий:  
  
-   Диаграмма поможет пользователям определить задачи, которые они могут работать в?  
  
-   Диаграмма позволяет пользователям прогноз последствия потенциальных изменений?  
  
-   Диаграммы упрощают обнаружение трендов и выявить закономерности?  
  
-   Диаграмма позволит принимать более обоснованные решения?  
  
-   Диаграмма поможет ответить на определенный вопрос, у пользователей могут возникать в данном контексте?  
  
#### <a name="general-rules-for-charts"></a>Общие правила для диаграмм  
  
-   Четко метки данных. Иллюстрации без объяснение не так просто просто картинки.  
  
-   Запустите осей с нуля, чтобы избежать наклон пропорции. Длина и строка размер строки, важные визуальные подсказки для понимания связей между точками данных.  
  
-   Создание диаграмм, не инфографика. Инфографика являются художественного представления данных, и их основной целью является повествования visual. Диаграммы можно (и следует быть визуально привлекательные, но позволить данные говорить сами за себя.  
  
-   Избегайте skeumorphism, графические гистограмм, hashmarks контрастность и другие изменения инфографикой.  
  
-   Не используйте объемных эффектов в качестве декоративного элемента. Используйте их только в том случае, если они действительно неотъемлемой частью возможность воспринимать информацию.  
  
-   Избегайте несколько линий и заливок, как более двух цветов может затруднить этот тип диаграммы для чтения и правильно интерпретировать.  
  
-   Не используйте диаграмму (или любой рисунке) единственным средством основные сведения о концепции или взаимодействия с данными. Это вызывает проблемы для пользователей с нарушениями зрения.  
  
-   Не используйте диаграммы как предварительный или декоративных элементов на странице. Другими словами Если диаграммы не добавить всех пользователей значение или справки решить проблему, не используйте его.  
  
### <a name="chart-types"></a>Типы диаграмм  
 Типы диаграмм, используемых в Visual Studio включают линейчатые диаграммы, графики, измененные круговой диаграммы, называется кольцевой диаграммы, или «кольцевой диаграммы,» временные шкалы, Точечная диаграмма (также называется «диаграммы кластера») графики и диаграммы Ганта. Каждый тип диаграммы полезно использовать для обмена данными свой тип сведений.

##  <a name="BKMK_OnObjectUI"></a> На объект пользовательского интерфейса и просмотр  
 Этот раздел предоставляет контекст для просмотра, также известные как peek представление кода, тип-объект пользовательского интерфейса, уникальные для Visual Studio.  
  
### <a name="overview"></a>Обзор  
  
-   Пользовательского интерфейса на объект следует предоставить пользователю Дополнительные сведения или интерактивность без отвлекать от их основных задач.  
  
-   Основной шаблон для пользовательского интерфейса-object в Visual Studio, называется «сведения о точке внимания».  
  
-   На объект пользовательского интерфейса в Visual Studio — встроенный или с плавающей запятой и либо устойчивыми или временной.  
  
    -   В представлении кода считывания, тип-объект пользовательского интерфейса в Visual Studio является встроенной и устойчивой.  
  
    -   CodeLens тип-объект пользовательского интерфейса в Visual Studio с плавающей запятой и временных  
  
 Основные сведения о работе часть кода или находить сведения об этом коде часто требуется для переключения контекста и перейти на другое содержимое или другой разработчик окна. Эти смены контекста может быть нарушают работу, так как пользователи могут потерять фокус над своей первоначальной задачей, если они покидают их главного окна. Кроме того получение исходного контекста обратно, что может быть сложно, особенно в том случае, если переключение windows вызвала их исходный код быть скрыт другой пользовательский Интерфейс.  
  
 На объект пользовательского интерфейса соответствует шаблону, называется «сведения о точке внимания». Эти сообщения, всплывающие окна и диалоговые окна предоставляют пользователям дополнительные соответствующие сведения, добавляющий уточнить отзыв или интерактивность без потери сосредоточиться на своих основных задач. Пользовательский Интерфейс-object примеры всплывающих окон, которые отображаются, когда пользователь наведет указатель их значок в области уведомлений, красной волнистой линией неправильно написанное слово и представление peek появились в Visual Studio 2013.  
  
### <a name="decision-points"></a>Точки принятия решений  
 В Visual Studio Чтобы использовать этот шаблон сведения о точке внимания несколькими способами. Выбор механизма вправо и ее реализации образом единообразного, предсказуемого важно для общего впечатления. В противном случае пользователю может быть предоставлен сложными или нестабильной интерфейс, в котором затрудняет фокуса ввода из самого содержимого.  
  
#### <a name="relationships-between-master-and-detail-content"></a>Связи между главных и подчиненных содержимого  
 Сведения о точке внимания, используется для отображения отношения между содержимым, что пользователь является уделила («главный» содержимого) и дополнительные связанные content (содержимое «detail»). В этом шаблоне детализации содержимого четко связанным с содержимым пользователь работает с и могут быть отображены близко к главной содержимого. Дополнительные сведения и данные, которые не могут быть сведены без перегрузке содержимого главной следовать другой шаблон, такие как окна инструментов.  
  
-   **Всегда** отображения содержимого сведений в непосредственной близости от содержимого главной страницы.  
  
-   **Всегда** убедитесь, что содержимое сведений по-прежнему позволяет пользователю уделять больше внимания содержимого главной. Часто для этого рекомендуется для отображения содержимого сведений как близко к главной содержимое можно. Это можно сделать, отображая содержимое детализации во всплывающем окне рядом с содержимым, master или обработав встроенное содержимое детализации под главной содержимого.  
  
-   **Никогда не** используйте сведения в момент перехода пользователя от содержимого главной внимания. Если пользователям требуется для доступа к содержимому детализации отдельно, предоставлять явное действие, которое дает пользователю возможность сделать это.  
  
#### <a name="design-details"></a>Подробные сведения о макете  
 Как только вы уверены, что пользовательский Интерфейс для объекта правильный выбор, существует четыре основных проектирования:  
  
1.  **Сохраняемость:** содержимое должно быть постоянными или временными?   
    Будет пользователей должна сохраняться видимость перенаправить или взаимодействия с данные? Или будет пользователям быстро Обзор на данных, затем перейдите к своей основной задаче?  
  
2.  **Тип содержимого:** содержимое будет информационное, готовые к работе или навигационных?   
    Пользователю нужны дополнительные сведения о содержимом master? Необходимо выполнить задачу, которая влияет на содержимое главной пользователя? Или пользователь должен быть направлены на другой ресурс?  
  
3.  **Тип индикатора:** индикатора окружения смысл?   
    Данные целом удобным способом, а не отображаются без перегрузке master содержимого?  
  
4.  **Жесты:** жесты, что будет использоваться для вызова неуправляемого кода и закрыть пользовательский Интерфейс?   
    Как пользователь подключить содержимое детализации и отправить его сейчас? Существует значение, при добавлении жестов, такие как закрепление для переключения между состояниями промежуточных и устойчивые?  
  
 Каждой из этих точек четыре принятия решений будет оказывать влияние на основные компоненты пользовательского интерфейса-object.  
  
### <a name="on-object-ui-components"></a>Компоненты пользовательского интерфейса на объект  
  
1.  Тип контейнера (докладчика содержимого)  
  
    -   Числа с плавающей запятой  
  
    -   Встроенная функция  
  
2.  Тип содержимого  
  
    -   Информация: данные, которые могут быть статическими или динамическими  
  
    -   Готовые к работе: команд, позволяющих изменить содержимого главной страницы  
  
    -   Навигационный: ссылки, ведут к другому окну или приложения, например MSDN  
  
3.  Жесты  
  
    -   Вызов  
  
    -   Увольнения  
  
    -   Закрепление  
  
    -   Другие действия  
  
4.  Модель сохраняемости и фиксации  
  
    -   Временный  
  
    -   Durable  
  
    -   Автоматический  
  
    -   По запросу  
  
5.  Окружения индикаторы (необязательно)  
  
    -   Подчеркивание волнистой линией  
  
    -   Значок смарт-тега  
  
    -   Другие показатели окружающей среды  
  
#### <a name="container-content-presenter-type"></a>Тип контейнера (докладчика содержимого)  
 Существует два основных варианта для представления содержимого точке внимания:  
  
1.  **Встроенный:** докладчика встроенной, например в представлении считывания, появившихся в редакторе кода Visual Studio 2013, освобождая место для нового содержимого переключившись существующего содержимого.  
  
    -   **Предпочтение** представить выступающий встроенные, если ожидается, что пользователям может понадобиться тратят значительное количество времени обращения к или взаимодействие с содержимым.  
  
    -   **Избегайте** выступающий встроенные, если ожидается, что пользователи будут посмотрите на данные можно представить, а затем продолжить их основных задач с минимальными нарушениями.  
  
2.  **С плавающей запятой:** плавающей докладчика располагается как можно ближе к выбранным содержимым, как можно, но не изменять расположение исходного содержимого. Можно применять различные стратегии, например, отображение плавающую панель содержимого через ближайшего доступных пробелы для выделенного символа.  
  
    -   **Предпочтение** плавающей выступающий, если ожидается, что пользователи будут посмотрите на данные можно представить, а затем продолжить их основных задач с минимальными нарушениями.  
  
    -   **Избегайте** плавающей выступающий, если ожидается, что пользователи будут тратят значительное количество времени обращения к или взаимодействие с содержимым при наличии.  
  
#### <a name="content-type"></a>Тип содержимого  
 Существует три основных типа содержимого, которое может отображаться внутри любого контейнера пользовательского интерфейса на объект. Любое сочетание этих типов данных могут быть отображены. Ниже приведены три типы.  
  
1.  **Информация:** большинство на объект контейнеры пользовательского интерфейса будет отображаться каким-либо информационное содержимое. Содержимое может представлять сведения о текущем состоянии среды, или он может представлять сведения о возможных будущих состояние среды. Например он может использоваться для показывая эффект от определенной команды, например рефакторинг, в существующий код.  
  
    -   **Всегда** использовать каноническое представление данных, которые можно отобразить. Например код должен выглядеть код, включающий подсветку синтаксиса и должны учитывать шрифта и другие параметры среды, которые пользователь задал.  
  
    -   **Всегда** рассмотреть возможность поддержки каких-либо действий над информационное содержимое, которое бы Если ту же информацию представляется в виде содержимого главной страницы. Например если презентация существующего кода в контейнер пользовательского интерфейса на объект, настоятельно рекомендуется поддержка возможности просмотра и изменения кода.  
  
    -   **Всегда** рекомендуется использовать другой цвет фона, если представления информационное содержимого, которое представляет возможных будущих состояние.  
  
2.  Готовые к работе: некоторые на объект пользовательского интерфейса контейнеры обеспечивают возможность выполнения некоторых операций над master содержимым, таких как выполнение операции рефакторинга.  
  
    -   **Всегда** положение пригодных к использованию команды отдельно от информационное содержимое.  
  
    -   **Всегда** включения и отключения действия, когда это необходимо.  
  
    -   **Всегда** ссылаться на стандартные правила для представления команд в диалоговые окна.  
  
    -   **Всегда** сохранить минимальное количество действий, предоставляемых в контейнер пользовательского интерфейса на объект для является абсолютным. Взаимодействие с пользовательским Интерфейсом-объект должен быть упрощенный, быстрый качества. Пользователь не следует расхода энергии на управлении сам контейнер пользовательского интерфейса на объект.  
  
    -   **Всегда** рассмотрим, как и когда контейнер пользовательского интерфейса на объект будет закрыт или отклонено. Рекомендуется любое действие, которое завершает диалог между основные и подробные содержимого следует закрывать контейнер пользовательского интерфейса на объект при вызове этого действия.  
  
3.  **Навигационные:** некоторые на объект пользовательского интерфейса контейнеры включают в себя ссылки, которые ведут к другому окну или приложения, например открытие статьи MSDN в веб-браузере пользователя.  
  
    -   **Всегда** добавить все навигационные ссылки с «Открыть», чтобы пользователи будут получать уведомления, куда выполняется переход некоторые другого содержимого.  
  
    -   **Всегда** разделения ссылок для перехода по ссылкам пригодных к использованию.  
  
#### <a name="ambient-indicators-optional"></a>Окружения индикаторы (необязательно)  
 Индикаторы окружения могут быть небольшие, включая текст, представленных в контрастных цветов от остальной части кода, или очевидно, включая символы tickler значков смарт-тег и подчеркивание волнистой линией. Индикаторы внешней информацией о доступности соответствующих дополнительных сведений. В идеальном случае они предоставляют полезные сведения, даже без необходимости пользователю взаимодействовать с ними.  
  
-   **Всегда** размещения индикатора окружения, таким образом, чтобы она не будет отвлекать внимание или перегрузки пользователя. Если пользователь не сможет положение индикатора окружения таким образом, рекомендуется другое решение.  
  
-   **Всегда** как можно ближе к содержимому, которое относится к позиции окружения индикатор.  
  
-   **Всегда** попытайтесь создать признаком того, что сводные сведения, которые он предоставляет. Подберите показатель числа доступных элементов данных (например, «3 ссылки» вместо просто «ссылки») или представить иным способом объединения данных.  
  
    -   В случаях, когда данные для индикатора не всегда быть вычисляется и отображается немедленно подберите прогрессивного отзывы как вычисляются значения. Например рассмотрим анимация изменения, которые отражают обновления, доступные данные примерно так, как живую плитку электронной почты на Windows Phone обновляет как числа непрочитанных сообщений электронной почты.  
  
-   **Никогда не** добавить дополнительные индикаторы, чем пользователя может занять достаточной степени для конкретной части содержимого. Индикаторы окружения должно быть полезным без необходимости вмешательства пользователя. Индикаторы утрате их окружением они требуют переполнения и другими элементами управления для их отображения.  
  
#### <a name="gestures"></a>Жесты  
 Ключевым аспектом разрешить пользователю Сфокусируйтесь на содержимого главной — с помощью поддержки правой жесты, чтобы открыть и закрыть содержимое дополнительных сведений.  
  
-   **Всегда** пользователю необходимо выполнить некоторые явный жест, чтобы открыть дополнительное содержимое. Следующие общие откройте жесты.  
  
    -   **При наведении указателя мыши:** подсказок или неинтерактивной информационное содержимое  
  
    -   **Явная команда:** встроенного докладчика  
  
    -   **Дважды щелкните индикатор окружения:** CodeLens всплывающего окна  
  
-   **Всегда** закрыть детализации содержимого при каждом нажатии клавиши Esc.  
  
-   **Всегда** учитывать контекст пользовательского интерфейса на объект. Для содержимого выступающий, которые допускают взаимодействие внутри контейнера внимательно рассмотрите вопрос для отображения дополнительных сведений при наведении курсора мыши, которая вероятнее всего прервет рабочего процесса пользователя.  
  
-   **Никогда не** отображения содержимого при наведении курсора мыши, которое отображается для редактирования или приглашает взаимодействия с пользователем. Это поведение может расстраивать планы пользователей, если они пытаются выполнить для перемещения курсора над содержимым сведений, как стандартное поведение подсказки немедленно прекратить, когда указатель мыши выходит за пределы главного содержимого, который его создал.  
  
##  <a name="BKMK_SelectionModels"></a> Выбор модели  
  
### <a name="overview"></a>Обзор  
 Модель выделения — это механизм, используемый для указания и подтверждения операции с одним или несколькими объектами интерес в пользовательском интерфейсе. В этом разделе обсуждаются выбора шаблоны взаимодействия внутри редакторов документа Visual Studio: текстовые редакторы, поверхности разработки и моделирования поверхности.  
  
 Пользователи должны иметь показывает, в Visual Studio, что они работают, и Visual Studio должен вернуть предсказуемо отзыв пользователям о том, что он работает. Различия или неправильным обменом данными между пользователем и пользовательского интерфейса может привести к пользователю, не отметить действие, которое может иметь непредвиденные последствия. Часто ошибки остается незамеченной, пока пользователь видит что-то отсутствует или был изменен. Выбор моделей, поэтому один из наиболее важных элементов пользовательского интерфейса. Выбор моделей в Visual Studio согласованы с Windows, имеют незначительные изменения.  
  
 В Visual Studio, как и в Windows Выбор моделей отличаются в зависимости от контекста, в котором происходит взаимодействие. Выбор может произойти в четырех типов объектов.  
  
-   Text  
  
-   Графические объекты  
  
-   Списки и деревья  
  
-   Таблицы  
  
 В этих объектах существует три типа компонентов:  
  
-   Непрерывный  
  
-   Раздельные  
  
-   Region  
  
#### <a name="scope"></a>Область  
 Самым важным компонентом выбора является обеспечение, что пользователю известен в окно, которое они работают (активация) и, задачей которого является найти (выделение). Visual Studio расширяет функциональные возможности управления окна в Windows, но схема активации одинаково: взаимодействия с окном перемещение фокуса в окне. В Visual Studio есть два индикаторы для активации: один для окон документов и один для окон инструментов.  
  
 Вкладка окна документа, поступающих на передний план и изменить цвет фона для окон документов, указывается активного окна:

 Для окон инструментов окна указывается изменение цвет для области заголовка окна инструментов:

 **Активное окно, отображающее основной выделенный объект узла**

#### <a name="context"></a>Контекст  
 Visual Studio была разработана для сохранения строгого понятием контекста, отслеживание которой работает пользователь. Только одно окно активен, является ли окно инструментов или документа. Однако окно переднего плана документа всегда сохраняет скрытые выделения. Несмотря на то, что фокус может быть в окне инструментов, окна документа, которая была активна отображает, даже в неактивном состоянии. Это позволяет сохранить контекст пользователя в документе, в которую они изменения, показывающая, что Visual Studio сохранил свое состояние, чтобы они могут возвращать и сдвиг беспрепятственно между окнами документов и окнами.  
  
### <a name="text-selection"></a>Выделение текста  
 Редакторы Visual Studio, которые являются строго текстовое, такие как встроенные текстового редактора, используйте ту же модель выделения текста и внешний вид, описанных в [и указатели мыши](https://msdn.microsoft.com/en-us/library/dn742466.aspx) Windows взаимодействия рекомендации по пользовательскому интерфейсу на странице MSDN. Фокус ввода в редакторе текста обозначается символом вертикальной черты, вызывается курсор. Позиция ввода находится один пиксель толстой и цветной в сумме все, что находится за его. Мигает, согласно задается скорость **частота мигания курсора** в **скорость** вкладке **клавиатуры** элемент панели управления.  
  
#### <a name="contiguous-and-disjoint-selection"></a>Выбор смежных и непересекающиеся  
 Только выделения в текстовом редакторе является непрерывным. Раздельные текста выбранных элементов не допускаются, но должны быть устранены в редакторах графический объект. При указателя мыши на область текста, вид курсора меняется в i-образный курсор. Одним щелчком помещается точка вставки в текстовом редакторе, по щелчку. Удерживая нажатой кнопку мыши начинается выделение и отпустить кнопку мыши заканчивается выделение.  
  
#### <a name="region-selection-box-selection"></a>Выбор региона (Выбор)  
 Visual Studio поддерживает выбор региона в текстовом редакторе, а это называется выделения. Выбор позволяет пользователю выбрать область текста, который не выполняет обычный текстовый поток. Как и в случае выбора стандартного текста, выбор должны быть смежными. Выбор инициируется, удерживая нажатой клавишу Alt при перетаскивании мышью. Также можно инициировать выделения, удерживая клавишу Alt и смены ключей с помощью клавиш со стрелками для указания области выделения. Выбор использует выделение обычным блоком выделения и показывает мигания в конце области выделения курсора точки вставки.

#### <a name="text-selection-appearance"></a>Внешний вид выделения текста  
 Можно настроить цвета, используемые для активных и неактивных выделения в редакторе. Чтобы настроить внешний вид редактора, пользователь может перейти к **Сервис > Параметры**и найдите **среды > шрифты и цвета > текстовый редактор**.  
  
### <a name="graphical-selection"></a>Выбор графического  
  
#### <a name="interaction"></a>Взаимодействие  
 Выбор графических объектов может быть сложным и зависит от нескольких факторов:  
  
-   **Модель редактора основной выделенный объект.** Редакторы, которые содержат графические объекты могут также использоваться для редактирования текста или таблицы. Например редактор может быть текстовый редактор также поддерживает размещение графических объектов, таких как конструктора Visual Studio XAML. Поддержка нескольких типов объектов может повлиять на том, как пользователь выбирает группы состоит из различных типов объектов.  
  
-   **Поддержка состояния основного и дополнительного выбора.** Редактор предоставляет выбор основного и дополнительного состояния, чтобы объекты можно редактировать в конфигурациях, выровнены друг относительно друга, размер вместе, и т. д.  
  
-   **Поддержка редактирования по месту.** Редакторы также можно разрешить содержимое их графических объектов, который нужно изменить. Например прямоугольник также может содержать текст внутри, которые могут быть изменены пользователем. Кроме того этот текст может быть по центру или по ширине. Редактирования по месту включает в себя более высокого уровня взаимодействия с пользователем и требует соответствующего набора визуальные подсказки для представления пользователю информации о состоянии.  
  
#### <a name="mouse-interaction"></a>Взаимодействие мыши  
  
|Входные данные|Результат|  
|-----------|------------|

|Щелкните выбранный объект|Активирует редактирования по месту, если объект поддерживает его. Щелкнув вне объекта деактивирует режима редактирования по месту.|  
|Дважды щелкните объект|Открывает кода программной части объект для редактирования и может вставить обработчик событий по умолчанию, если это необходимо.|  
|Указывает на объект|Изменяет указатель на указатель перемещения. Вид объекта, например к яркость или цвет, может измениться.|  
|Установите указатель на маркер выбора|Изменяет указатель на курсор изменения размера. Для объектов, которые поддерживают поворот некоторые маркеры выделения может изменяться указатель на поворот курсор указателя по-разному (например, переместить дальше) по отношению к маркер выделения.|  
|Перетаскивание|Даже если объект ранее не выбран, изменяет указатель на указатель перемещения и перемещение объекта.|  
|Редактор теряет фокус|Отключает режим редактирования по месту, несмотря на то, что объект сохраняет содержимое и внешний вид, что и во время последнего состояния операции или выбора.|  
|Выбор объектов|Указывает границы, пунктирная линия или других визуально distinct обработки, чтобы выделить границы объекта.|  
|Изменение размера выбранного объекта|Указывает маркеры выделения.<br /><br /> Изменить его размер имеет восемь маркеров, представляющих каждого направления, в котором их можно изменять. Меньшее число дескрипторов может использоваться, если объект можно изменять только в определенных направлениях. Когда пользователь изменяет размер объекта до которых восемь маркеров бы интерактивный, может использовать четыре маркера. Дескриптор размеры подключен к метрики границы и границей окна с **GetSystemMetrics** функции API для размера пропорционально разрешение экрана.<br /><br /> ![Маркеры изменения размера](../../extensibility/ux-guidelines/media/0713-05_resizehandles.png "0713 05_ResizeHandles")|  
|Поворот выделенного объекта|![Маркеры вращения](../../extensibility/ux-guidelines/media/0713-06_rotate.png "0713 06_Rotate")|  
  
#### <a name="keyboard-interaction"></a>Взаимодействие с клавиатурой  
  
|Входные данные|Результат|  
|-----------|------------|  
|Tab|Перемещает фокус индикатор среди логический порядок объектов в редакторе. Это может быть слева направо и сверху вниз в зависимости от **TabIndex** (или эквивалентную) значение свойства, порядок создания объектов и редактора общего назначения. Shift + Tab меняет направление индикатора фокус.|

|Сочетание клавиш Ctrl или SHIFT|Добавляет объект к группе выбора. CTRL можно удалить объекты по отдельности из выделенной области группы.|

|Ctrl + Shift + клавиши со стрелками|Изменяет размер выбранных объектов в соответствующих направлении, большими шагами (например, 10 точек за раз)|  
  
 Когда пользователи изменяют механизмов, смысл для объектов, для автоматического изменения размера с вводимыми пользователем данными. Например если пользователь изменяет элемент управления label, метка увеличивался для отображения текста, достаточно ввести пользователь. Если этого не сделать, пользователь должен изменять размеры элемента управления вручную после редактирования текста. Если у пользователя имеется много элементов управления, оно становится механических и недоступными задачи.  
  
#### <a name="graphical-containers"></a>Графические контейнеры  
 В некоторых случаях графических редакторах являются контейнерами для других графических объектов, такие как элемент управления панели Windows Forms или управления макет сетки в конструкторе HTML. Если ваш редактор предоставляет контейнеры для других графических объектов, следующие модели выбора следует использовать только контейнера (объекты в контейнере следующим стандартной модели, как описано выше):  
  
|Входные данные|Результат|  
|-----------|------------|  
|Одним щелчком в контейнере|Выделяет объект контейнера без непосредственно при выборе любого из содержащихся в них объектов. Контейнер может перемещении и изменении размеров стандартных действий мыши и клавиатуры (как описано выше). Содержащиеся объекты перемещаются по отношению к контейнеру, но содержащиеся объекты не изменяются, если не выбраны также напрямую.|  
|Наведите указатель мыши на границу области контейнера|Преобразует указатель мыши в курсор move, указывающее, что контейнер можно переместить.|  
|Перетащите область границы контейнера|Изменение указателя мыши для перемещения курсора и перемещает контейнер (и объекты, содержащиеся в пределах). Контейнер не может быть перемещен без предварительного выбора одним щелчком.|  
|Щелчком на объект в контейнере|Отмена контейнера (если выбраны) и выбирает только объект, которую щелкнул пользователь.|  
|+ Нажмите клавиши Ctrl или SHIFT, щелкните на содержащегося объекта или контейнера|Добавляет объект пользователь щелкнул выделенной области или группе выбора. Если пользователь щелкнул объект уже является членом группы выбора, он удаляется из группы выбора.|

#### <a name="tree-view"></a>Иерархическое представление  
 Выбор в виде дерева показан простой цветом. Если пользователь щелкает значок узла или имени узла, узел выделяется. Треугольное глифы слева от узла разверните или свернуть дерево, но не влияют на выбор пользователя, за одним исключением: от родительского узла разрушения Выбор дочернего узла, превратиться в родительский объект.  
  
 ![Стандартные представления в Visual Studio](../../extensibility/ux-guidelines/media/0713-25_treeview.png "0713 25_TreeView")  
  
 **Стандартные представления в Visual Studio**  
  
 В древовидном представлении может поддерживать выбранные смежных и раздельных, даже на нескольких уровнях дерева. Смежных или раздельных множественный выбор, которые должны вноситься на узлы дерева видимым. При сворачивании узла раздельных выделение теряется, и узел, который был свернут получает выделение. Таким образом пользователь может видеть узлы, которые будут затронуты операции. При свертывании узлов становится ясно какие узлы могут отразиться.  
  
 При выборе родительского узла, следует применить операцию в родительский объект в тех случаях, когда имеет смысл для операции для применения к родительской и всех его дочерних элементов. В этом случае укажите дополнительный пользовательский Интерфейс во время операции, такие как флажок или диалоговое окно подтверждения, чтобы сделать явную параметр «применять ко всем дочерним» для пользователя.  
  
##### <a name="renaming"></a>Переименование  
 Если узлы в дереве поддерживает переименование, переименование должно выполняться на месте. Операции с заменой должны быть стандартные среди всех элементов управления "дерево" в Visual Studio. Введите команду переименования, тут же активирует режим редактирования по месту текста выделение, охватывающие полное имя узла, готов к приему ввод данных пользователем. Если узел представляет файл, имя файла должно содержать расширение. Выделение должно включать имя файла и расширение не только из тела.  
  
|Входные данные|Результат|  
|-----------|------------|  
|ВВОД - клавиша|Фиксирует операции переименования|  
|Клавиша ESC|Отменяет операцию переименования|  
|Щелкните вне области редактирования по месту|Фиксирует операции переименования|  
|Отменить|Обеспечить возможность легко отмены для отмены операции переименования|  
  
#### <a name="selection-within-lists-and-grid-controls"></a>Выбор в списках и элементам управления DataGrid  
 Ключевое понятие в списке выбора является, на основе строк, это значит, что при выделении строку целиком установлен как единое целое. В отличие от этого таблицы можно разрешить отдельным ячейкам, необходимо выбрать без влияния на другие аспекты строки. Таблицы могут также содержать иерархию вложенных строк (например, в представления в виде дерева), позволяющие всей ветви иерархии выделить и потребуется снять этот флажок, взаимодействуя с родительских строк. Выбор в списках отображаются с простого выделения цветом на всю строку данных. Фокус показана пунктиром пиксель вокруг текущей редактируемой строки или ячейки (строка, если все ячейки доступны только для чтения).  
  
> [!NOTE]
>  **Фокус** и **выбора** представляют собой разные понятия. *Фокус* указывает на то, какие пользовательского интерфейса элемент предназначена для получения входных данных, явно не направлены на другой объект, тогда как *выбора* относится к состоянию объекта включения в набор объектов, на котором последующие операции могут выполняться.  
  
 Выбранные элементы в списки могут быть смежными, несвязанными, или регион. Когда множественный выбор разрешен, смежных и раздельных выбора следует всегда будет поддерживаться, во время поддержка Выбор региона (поле) является необязательным. Выбор региона инициируются путем перетаскивания в пустом месте тела списка.  
  
|Object|Выбранное|  
|------------|---------------|  
|Список|Непрерывный|Всегда поддерживается (если множественный выбор разрешен).|  
|Список|Раздельные|Всегда поддерживается (если множественный выбор разрешен).|  
|Список|Region|Дополнительная поддержка. Начата, перетаскивая указатель мыши в пустом месте тела списка.|  
  
 Щелкните один раз на списке выбирает строки, в котором произошел щелчок. Если пользователь щелкнет щелкните ячейку списка с поддержкой редактирования по месту, ячейки также немедленно активируется для встроенного редактирования. В противном случае строку целиком немедленно выбран и отображается выделения.  
  
 Перетаскивание в теле список выполняет одно из трех действий:  
  
-   Инициирует Выбор региона, если список поддерживает его и кнопок мыши пустое пространство  
  
-   Инициирует операцию перетаскивания, если список ячейку или строку поддерживает выполняется источник перетаскивания  
  
-   Выделение текущей строки  
  
##### <a name="in-place-editing"></a>Внутреннее соединение  
 Если разрешен редактирования по месту, существует две основные модели: Выбор простого редактирования элемента управления и свойства. С помощью простой элемент управления содержимое выделяется и становится готова для ввода данных сразу же после редактирования по месту активируется пользователем. Там, где реализован Выбор свойства кнопки, которая вызывает выбор свойства отображается после активации режима редактирования по месту и текущего выделения не выделяется. Кнопка выбора следует выравниваются по правому краю ячейки. Примеры редактирования по месту, в разделе **окно свойств** и **список задач** в Visual Studio.  
  
##### <a name="keyboard-support"></a>Поддержка клавиатуры  
 Поддержка клавиатуры для выбора в списки и таблицы следует стандартным соглашениям Windows:  
  
-   Клавиши со стрелками перехода по списку, выбрав каждой строки или ячейки, при перемещении фокуса.  
  
-   SHIFT + клавиши со стрелками выполняет связанному выделению в направлении клавиш со стрелками.  
  
-   CTRL + стрелка, следуют переключает пробел между Добавление и удаление элементов списка из выделенной области, Создание несвязанного выделения.  
  
-   Для таблицы, которые содержат вложенные иерархии клавишу со стрелкой вправо разворачивает родительской строки и клавишу со стрелкой влево сворачивает один.  
  
-   Фокус перемещается клавишей Tab между ячейками в текущей строке ячеек, для редактирования.  
  
-   Клавиши ВВОД выполняет команду по умолчанию для элемента в списке (часто **откройте**).  
  
-   Клавиша F2 активирует встроенного редактирования для текущей выбранной ячейки.

### <a name="window-configurations"></a>Конфигурации окон  
 Конфигурация окна — это основные представления среды разработки — это схемы, состоящее из списка окон инструментов присутствует и упорядочения. Для windows, управляемых в интегрированной среде разработки (IDE windows) сведения о макете сохраняется на пользователя, когда пользователь запускает интегрированную среду разработки, макет окна появится таким же, как действуют при выходе из Visual Studio. Состояние и положение окон интегрированной среды разработки, сохраняются в файле пользовательских параметров в формате XML. Окна инструментов, созданные пакеты, загруженный в IDE сохранения информации о состоянии в реестре и могут находиться не на пользователя.  
  
#### <a name="profile-specific-layouts"></a>Макеты определенного профиля  
 Каждый профиль включает средство макетов окон, организованных образом знакома конкретного разработчика лица (разработчиков Visual C++ ожидать **обозревателе решений** левой части интегрированной среды разработки, а ожидатьразработчиковC# **Обозреватель решений** справа). Макеты окон определенного профиля загружаются после выбора профиля во время запуска. Автор пакета следует определить макет окна, наилучшим образом подходят для их улучшения качества, зная, что изменения, вносимые пользователем в конфигурации окна будут сохраняться.  
  
##  <a name="BKMK_TouchInput"></a> Сенсорный ввод  
 Пользователи все чаще используются разработки продуктов корпорации Майкрософт на сенсорных устройствах. Однако существуют препятствия, которые трудно использовать средства разработки на сенсорных устройствах. Пользователи ожидают, что наши продукты для взаимодействия с надежным и точным сенсорный ввод. Цель данного руководства — принять решение о возможности, которые сенсорного ввода для внедрения и чтобы стимулировать согласованное сенсорного между Visual Studio и связанных продуктов.  
  
### <a name="levels-of-experience"></a>Уровни качества  
 Следующие уровни качества предназначены для использования в качестве руководства помогают командам решить возможностями сенсорного ввода, предложить на основе их нужного уровня инвестиций интерес свяжется с вами.  
  
-   **Основные возможности** для групп, которым требуется предоставить touch возможности, поэтому нет концов сообщений во всей своей работы.  
  
-   **Оптимизированными качества** — для команд, которые могут обеспечить наиболее обычные сенсорные возможности (например, которые обычно доступны в Интернет-приложения браузера).  
  
-   **Повышением качества** для команд, которые хотите добавить такие возможности как жесты или других дополнительных возможностей, которые могут создавать свои приложения сенсорных понятное.

#### <a name="gestures"></a>Жесты  
 Жесты предоставляют пользователям ярлык для команд, которые в противном случае может потребоваться более сложный взаимодействия. Ознакомьтесь с Windows рекомендации по [обычные сенсорные жесты для приложения для настольных компьютеров](http://msdn.microsoft.com/en-us/library/windows/desktop/dd940543\(v=vs.85\).aspx)и следуйте инструкциям в этом руководстве для большинства жестов, включая простые жесты, например панорамированием и масштабированием.
