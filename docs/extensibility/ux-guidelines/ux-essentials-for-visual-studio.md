# <a name="ux-essentials-for-visual-studio"></a>Основы UX для Visual Studio
## <a name="best-practices"></a>Рекомендации  
  
### <a name="1-be-consistent-within-the-visual-studio-environment"></a>1. Быть согласованы в среде Visual Studio.  
  
-   Выполните существующие шаблоны взаимодействия в оболочке.  
  
-   Спроектировать для соответствия требованиям оболочки visual язык и мастерства.  
  
-   Использование общих команды и элементы управления, если они существуют.  
  
-   Понимание иерархии Visual Studio и как он устанавливает контекст и диски пользовательского интерфейса.  
  
### <a name="2-use-the-environment-service-for-fonts-and-colors"></a>2. Используйте службу среды для шрифтов и цветов.  
  
-   Пользовательский Интерфейс должен подчиняться параметр текущего шрифта среды, пока она предоставляется для настройки на странице "шрифты и цвета" в диалоговом окне Параметры.  
  
-   Элементы пользовательского интерфейса, необходимо использовать службу VSColor, с помощью общей среды маркеры или маркеры определенным функциям.  
  
### <a name="3-make-all-imagery-consistent-with-the-new-vs-style"></a>3. Согласовать все изображения с нового стиля VS.  
  
-   Выполните принципы проектирования Visual Studio значки, глифы и другие рисунки.  
  
-   Не следует помещать текст в графических элементов.  
  
### <a name="4-design-from-a-user-centric-perspective"></a>4. Разработка с точки зрения ориентированных на пользователя.  
  
-   Создайте поток задач до отдельных компонентов внутри него.  
  
-   Ознакомиться с пользователями и предоставляют эти сведения в явном виде в своей спецификации.  
  
-   При рассмотрении пользовательского интерфейса, оцените полные возможности, а также подробные сведения.  
  
-   Разрабатывать интерфейс пользователя, чтобы обеспечить его функциональных возможностей и привлекательный независимо от языкового стандарта или языка.  
  
## <a name="screen-resolution"></a>Разрешение экрана  
  
### <a name="minimum-resolution"></a>Минимальное разрешение  
 Минимальное разрешение для Visual Studio Dev14 — 1280 x 1024. Это означает, что *возможных* в Visual Studio это разрешение, хотя он может быть оптимального взаимодействия с пользователем. Нет никакой гарантии, что все аспекты будет использовать при разрешении меньше 1280 x 1024.  
  
 Размер начального диалогового окна не должна превышать 1000 пикселов в высоту таким образом, чтобы уместить внутри фрейма интегрированной среды разработки в рамках этого минимальное разрешение 96 точек на дюйм.  
  
### <a name="high-density-displays"></a>Отображает высокой плотности  
 Пользовательский Интерфейс в Visual Studio должен хорошо работают в всех точек на ДЮЙМ, поддерживаемых Windows изначально коэффициенты масштабирования: 150%, 200% и 250%.  
  
## <a name="anti-patterns"></a>Антишаблоны  
 Visual Studio содержит множество примеров пользовательского интерфейса, выполните наши правила и рекомендации. В целях согласованными разработчики часто заимствуют созданные из примерно то, что они создают шаблоны проектирования пользовательского интерфейса продукта. Несмотря на то, что это хороший подход, что помогает нам диска взаимодействия с пользователем и визуального проектирования, в некоторых случаях поставки компонентов с несколько сведения, которые не соответствуют нашим рекомендациям из-за ограничений расписания или исключить приоритетов. В этих случаях не будет команды копирования одной эти «антишаблоны», так как они распространением неверный или несогласованный пользовательского интерфейса в среде Visual Studio.  
  
### <a name="required-fieldssettings-shown-in-error-state-by-default"></a>Необходимые поля и параметры по умолчанию отображаются в состоянии ошибки  
  
#### <a name="feature-team-goals"></a>Цели команды  
  
-   Предупредить пользователей о том, что они добавлены элемент, который необходимо настроить.  
  
-   Привлечь внимание пользователя к области, которые необходимо входных данных.  
  
#### <a name="anti-pattern-solution"></a>Шаблон решения  
 Как только пользователь инициализировал действие и прежде, чем они выполнили задачу, немедленно разместите значки критической рядом с области, которые требуется настройка.  
  
#### <a name="example-manifest-designer-declarations"></a>Пример: Объявления конструктора манифестов  
 Добавить объявление в список сразу помещается в состояние ошибки, который продолжает существовать, пока пользователь задает необходимые свойства.  
  
 В этом случае существует дополнительную проблему, потому что содержит значок, используемый для предупреждения «x», чтобы удалить общий значок не может использоваться рядом с ним. В результате пользовательский Интерфейс используется кнопка Remove, более неуклюжим элемента управления.  
  
 ![Манифеста конструктор антишаблон объявления об ошибке](../../extensibility/ux-guidelines/media/manifestdesignererrordeclarationsanti-pattern.png "ManifestDesignererrordeclarationsanti шаблон")  
  
 **Размещение пользовательского интерфейса в состоянии ошибки по умолчанию является шаблон Visual Studio.**  
  
#### <a name="alternatives"></a>Альтернативные варианты  
 Более удачное решение этой проблемы может быть:  
  
-   Позволяет пользователю добавить объявление без предупреждения, а затем переместите немедленно для задания свойств элемента.  
  
-   Добавить значок предупреждения (золотой треугольник) при перемещении фокуса от элемента, например, чтобы добавить другое объявление в список или пытаться изменить вкладок в конструкторе.  
  
-   Если пользователь пытается изменить вкладки перед заданием свойства на все объявления, pop диалоговое окно, объясняющее, что приложение не будет создавать (или любые последствия) пока предупреждения не будут разрешены. Если пользователь закрывает диалоговое окно и изменяет вкладки в любом случае затем значок (критическое или предупреждение, соответствующим образом) добавляется на вкладку «объявления».  
  
### <a name="forcing-the-user-to-read-text-before-dismissing-ui"></a>Принудительное пользователям читать текст перед закрытием пользовательского интерфейса  
  
#### <a name="feature-team-goals"></a>Цели команды  
 Не позволяет пользователю закрыть пользовательский Интерфейс без первого просмотра текст пояснения.  
  
#### <a name="anti-pattern"></a>Анти-шаблон  
 Команда вставки ссылки на видео в различных областях пользовательского интерфейса VS, решил общий шаблон X закройте объяснение кнопки и подсказка в соответствии с UX и реализован в раскрывающемся списке и ссылку на «Больше не показывать».  
  
 ![Антишаблон пояснительного текста — неправильный](../../extensibility/ux-guidelines/media/incorrectuseofmultipleclicks.png "Incorrectuseofmultipleclicks")  
  
 **Неверный: принудительное пользователям читать пояснительного текста перед закрытием пользовательского интерфейса является шаблон в Visual Studio.**  
  
#### <a name="result"></a>Результат  
 Вместо простого кнопку Закрыть (одним щелчком) пользователь вынужден позволяет просто закрыть пользовательский Интерфейс везде, ссылки на видео отображаются двумя отдельными щелчками.  
  
#### <a name="alternatives"></a>Альтернативные варианты  
 Правильный подход для данной ситуации было бы выполнить шаблонов, типичных для Internet Explorer, Office и Visual Studio: при наведении курсора мыши, пользователь может видеть описание элемента tooltip и одним щелчком мыши скрывает пользовательский Интерфейс.  
  
 ![Антишаблон пояснительного текста — правильный](../../extensibility/ux-guidelines/media/explanatorytextanti-pattern-correct.png "Explanatorytextanti шаблон исправление")  
  
 **Правильное: как было задумано, видео ссылки должны отображаться подсказка с дополнительными сведениями, при наведении курсора мыши и нажав кнопку «X» следует закрыть сообщение без необходимости взаимодействовать с дальнейшей.**  
  
### <a name="using-command-bars-for-settings"></a>Использование командной строки для параметров  
 ![Антишаблон командной-рис](../../extensibility/ux-guidelines/media/commandbaranti-pattern-figurea.png "Commandbaranti шаблон FigureA")  
  
 **Рисунок а антишаблон командной**  
  
 **Рисунок** представляет этот шаблон: размещение параметр под кнопки, применяется не только в команду. В этот эскиз есть команды помимо начать отладку, как представление в браузере, запуск без отладки и шаг с заходом —, будут использовать выбранный параметр.  
  
 ![Антишаблон командной-рис. Б](../../extensibility/ux-guidelines/media/commandbaranti-pattern-figureb.png "Commandbaranti шаблон FigureB")  
  
 **Рисунок б. лучше, но по-прежнему антишаблон командной**  
  
 Немного лучше, но по-прежнему нежелательным, помещает параметры этого типа в панели инструментов, как показано в **рис. Б**. Хотя кнопки разбиения занимают меньше места и, следовательно, являются улучшение через раскрывающееся меню, оба проекта по-прежнему используется панель инструментов для продвижения то, что команда не.  
  
 ![Команда антишаблон — рисунок С](../../extensibility/ux-guidelines/media/commandbaranti-pattern-figurec.png "Commandbaranti шаблон FigureC")  
  
 **Рисунок C: правильно использовать шаблон панель команд Visual Studio**  
  
 В **рисунок С**, параметр привязан к последовательности команд. Нет глобального параметра устанавливается, и мы просто переключении между четырьмя командами. Это единственный случай, в котором допустимы команды на панели инструментов.  
  
### <a name="control-anti-patterns"></a>Элемент управления антишаблоны  
 Некоторые антишаблоны являются просто неправильного использования представления элемента управления или группы элементов управления.  
  
#### <a name="underlining-used-as-a-group-label-not-a-hyperlink"></a>Подчеркивание, используется в качестве метки группы, не гиперссылки  
 Подчеркивание текста следует использовать только для гиперссылки.  
  
 **Неверный:**  
  
 ![Подчеркивание антишаблонов в групповых метках](../../extensibility/ux-guidelines/media/0102-g_grouplabelincorrect.png "0102 g_GroupLabelIncorrect")  
  
 **Подчеркнутый текст, который не является гиперссылкой является шаблон Visual Studio.**  
  
 **Хороший:**  
  
 ![Подчеркивание антишаблонов в групповых метках (правильное)](../../extensibility/ux-guidelines/media/0102-h_grouplabelcorrect.png "0102 h_GroupLabelCorrect")  
  
 **Правильно оформленного, не гиперссылки текст отображается без каких-либо шрифт среды разработки.**  
  
#### <a name="clicking-on-a-check-box-results-in-a-pop-up-dialog"></a>Щелкнув флажок результаты во всплывающем диалоге  
 Щелкнув флажок «Включить удаленный рабочий стол для всех ролей» в мастере «Опубликовать приложение Windows Azure» немедленно появится всплывающее диалоговое окно порочной моделью Visual Studio. Кроме того, поле флажка не заполняет флажок После выбора, другой шаблон взаимодействия.  
  
 ![Антишаблон всплывающего окна флажка](../../extensibility/ux-guidelines/media/0102-i_checkboxpopup.png "0102 i_CheckboxPopup")  
  
 **Открывая диалоговое окно после нажатием флажков является шаблон Visual Studio.**  
  
### <a name="hyperlink-anti-patterns"></a>Антишаблоны гиперссылки  
 Следующий пример содержит два антишаблоны.  
  
1.  Переднего плана при наведении курсора мыши Включение red означает, что правильный цвет общей службой шрифт не используется.  
  
2.  «Дополнительные сведения» не соответствующий текст для ссылки на раздел общих понятий. Цель пользователя — не подробнее, их лучше понимать последствия по своему выбору.  
  
 ![Антишаблоны гиперссылки](../../extensibility/ux-guidelines/media/0102-j_hyperlinkincorrect.png "0102 j_HyperlinkIncorrect")  
  
 **Пропуск службы цвет и с помощью «Дополнительные сведения» для гиперссылки являются неэффективные шаблоны Visual Studio.**  
  
 **Лучшие решения:** представляют вопрос, щелкнув ссылку бы спросить пользователя.  
  
-   Как работают службы Windows Azure?  
  
-   Если необходим проект мобильных служб Windows Azure?  
  
#### <a name="using-click-here-for-links"></a>С помощью «Щелкните здесь» для ссылок  
 Гиперссылки должно быть описательные. Это порочной моделью использовать «Щелкните здесь» или любой аналогичные вариации.  
  
 **Плохо:** «Щелкните здесь для получения инструкций о том, как создать новый проект.»  
  
 **Хорошее:** «Как создать новый проект?»
