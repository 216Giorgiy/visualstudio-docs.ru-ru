##  <a name="BKMK_CommonControls"></a> Общие элементы управления  
  
### Обзор  
 Стандартные элементы управления составляют большую часть пользовательского интерфейса в Visual Studio. Наиболее распространенные элементы управления, используемые в интерфейсе Visual Studio необходимо следовать [правила взаимодействия с рабочего стола Windows](https://msdn.microsoft.com/library/windows/desktop/dn742399.aspx). Этот документ относится к Visual Studio и рассматриваются особые ситуации или сведения, которые расширить эти рекомендации Windows.

#### Визуальный стиль  
 В первую очередь необходимо учитывать при стили элементов управления является использование элементов управления в пользовательском Интерфейсе темой. Элементы управления в стандартный пользовательский Интерфейс не тематические пользовательского интерфейса и должны следовать [Обычный стиль рабочего стола Windows](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742399\(v=vs.85\).aspx), означает, что они не являются re шаблонный должны появиться в их внешний вид элемента управления по умолчанию.  
  
-   **Диалоговые окна standard \(служебная программа\):** не темой. Сделать не re шаблон. Используйте значения по умолчанию стиль базового элемента управления.  
  
-   **Инструментов, редакторы документа, поверхности разработки и тематические диалоговых окон:** использовать специализированные тематические внешний вид с помощью службы цвет.  
  
###  <a name="BKMK_Scrollbars"></a> Полосы прокрутки  
 Полосы прокрутки должны следовать [Общие шаблоны взаимодействия для полосы прокрутки Windows](https://msdn.microsoft.com/en-us/library/windows/desktop/bb787527\(v=vs.85\).aspx) если только они будут дополнены сведения о содержимом, например в редакторе кода.  
  
###  <a name="BKMK_InputFields"></a> Поля ввода  
 Поведение типичного взаимодействия выполните [рекомендации рабочего стола Windows для текстовых полей](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742442\(v=vs.85\).aspx).  
  
#### Визуальный стиль  
  
-   Поля ввода должны стиль не в диалоговых окнах служебной программы. Используйте основной стиль, встроенные в элемент управления.  
  
-   Тематические поля ввода должен использоваться только в тематические диалоговые окна и окна инструментов.  
  
#### Специальные взаимодействия  
  
-   Поля только для чтения будет иметь фон серого \(отключено\) но \(активный\) цвет переднего плана.  
  
-   Необходимые поля должны иметь **\< необходимые \>** как водяные знаки в них. Не изменяйте цвет фона только в редких случаях.  
  
-   Ошибка проверки: см. [Уведомления и хода выполнения для Visual Studio](../../extensibility/ux-guidelines/notifications-and-progress-for-visual-studio.md)  
  
-   Поля ввода должны иметь соответствующий размер в соответствии с содержимым не по ширине окна, в котором они отображаются, а также на произвольно соответствует длине длинные поля, например путь. Длина может указывать пользователю ограничений на количество знаков в поле допускаются.  
  
     ![Incorrect input field control width](../../extensibility/ux-guidelines/media/0707-01_incorrectinputfieldcontrol.png "0707\-01\_IncorrectInputFieldControl")   
     **Неверное поле ввода длины: маловероятно, что имя будет такой длины.**  
  
     ![Correct input field control width](../../extensibility/ux-guidelines/media/0707-02_correctinputfieldcontrol.png "0707\-02\_CorrectInputFieldControl")   
     **Исправление длины поля ввода: поле ввода является разумным ширину ожидаемое содержимое.**  
  
###  <a name="BKMK_ComboBoxesAndDropDowns"></a> Поля со списком и раскрывающиеся списки  
 Поведение типичного взаимодействия выполните [правила рабочего стола Windows раскрывающиеся списки и поля со списком](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742404\(v=vs.85\).aspx).  
  
#### Визуальный стиль  
  
-   В диалоговых окнах служебной программы выполнять не re шаблон элемента управления. Используйте основной стиль, встроенные в элемент управления.  
  
-   Тематические пользовательского интерфейса, поля со списком и раскрывающиеся следуйте стандартной темы для элементов управления.  
  
#### Макет  
 Поля со списком и раскрывающиеся должны иметь соответствующий размер в соответствии с содержимым не по ширине окна, в котором они отображаются, а также на произвольно соответствует длине длинные поля, например путь.

###  <a name="BKMK_CheckBoxes"></a> Флажки  
 Поведение типичного взаимодействия выполните [рекомендации рабочего стола Windows для флажков](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742401\(v=vs.85\).aspx).  
  
#### Визуальный стиль  
  
-   В диалоговых окнах служебной программы выполнять не re шаблон элемента управления. Используйте основной стиль, встроенные в элемент управления.  
  
-   Тематические пользовательского интерфейса флажки следуйте стандартной темы для элементов управления.  
  
#### Специальные взаимодействия  
  
-   Взаимодействие с флажком никогда не должны Извлеките диалоговое окно или перейти в другую область.  
  
-   Выравнивание флажка с базовой линии первой строки текста.  
  
     ![Incorrect check box alignment](../../extensibility/ux-guidelines/media/0707-05_incorrectcheckboxalign.png "0707\-05\_IncorrectCheckBoxAlign")   
     **Неправильное выравнивание флажка: флажок ориентирован на текст.**  
  
     ![Correct check box alignment](../../extensibility/ux-guidelines/media/0707-06_correctcheckboxalign.png "0707\-06\_CorrectCheckBoxAlign")   
     **Исправить выравнивание флажка: флажок выравнивается по базовой линии первой строки текста.**  
  
###  <a name="BKMK_RadioButtons"></a> Переключатели  
 Поведение типичного взаимодействия выполните [рекомендации рабочего стола Windows для переключателей](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742436\(v=vs.85\).aspx).  
  
#### Визуальный стиль  
 В диалоговых окнах служебной программы выполнять не стиль переключателей. Используйте основной стиль, встроенные в элемент управления.  
  
#### Специальные взаимодействия  
 Используйте рамки группы, чтобы вложить варианты радио необязательно.  
  
###  <a name="BKMK_GroupFrames"></a> Группа кадров  
 Поведение типичного взаимодействия выполните [рабочий стол Windows рекомендации для группы кадров](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742405\(v=vs.85\).aspx).  
  
#### Визуальный стиль  
 В диалоговых окнах служебной программы выполнять не стиль рамки группы. Используйте основной стиль, встроенные в элемент управления.  
  
#### Макет  
  
-   Используйте рамки группы, чтобы включить радио варианты, если необходимо поддерживать различие группы в макете тесной необязательно.  
  
-   Никогда не используйте рамки группы для одного элемента управления.  
  
-   Иногда допустимо использовать горизонтальную линию вместо контейнера рамки группы.  
  
##  <a name="BKMK_TextControls"></a> Текстовые элементы управления  
  
### Подписи  
  
#### Состояние Active метки  
  
##### Диалоговые окна программы \(standard\)\)  
  
-   Как правило следуйте инструкциям Windows Desktop для метки элемента управления.  
  
-   В диалоговых окнах служебной программы должны надписи не полужирный, в стандартную среду шрифт и цвет текста. См. раздел [Шрифты и форматирование для Visual Studio](../../extensibility/ux-guidelines/fonts-and-formatting-for-visual-studio.md).  
  
-   Многоточие всегда должны следовать метки.  
  
##### Подпись \(тематические\) диалоговых окон\)  
 Элементы управления Label может быть полужирным шрифтом или светло\-серый.  
  
#### Состояние отключенного метки  
 Метки должны отражать внешний вид элемента управления, связанные с ними. Например при отключении связанного элемента управления метка следует появятся серым и отключен. Это требует никакой специальной обработке и обычно обрабатывается Операционной системой.  
  
#### Динамические метки  
 Динамические метки изменяется в зависимости от текущего выделения. По возможности используйте динамические метки в макетах основной\/подробности, чтобы помочь пользователю понять, что отображается информация относится к выбору конкретных элементов и не общие сведения.  
  
 ![Dynamic label used with dynamic content](../../extensibility/ux-guidelines/media/070702-01_dynamiclabel.png "070702\-01\_DynamicLabel")  
  
 **Пример динамические метка, используемая с динамическим содержимым**  
  
#### Пояснительный текст  
 Некоторые элементы интерфейса преимущества пояснительный текст Чтобы помочь пользователю понять назначение пользовательского интерфейса или указать, какое действие следует предпринять.  
  
-   Пояснительный текст в верхней части диалоговых окон, чаще всего, но может использоваться в других областях, чтобы предоставить инструкции для группирования сложного элемента управления.  
  
-   Пояснительный текст не является интерактивным, но может содержать ссылки на разделы справки.  
  
-   Используйте пояснительный текст экономно и только при необходимости.  
  
##### Форматирование  
 Пояснительный текст следует шрифт среды, стандартный элемент управления \(не тематические\) текста. См. раздел [Шрифты и форматирование для Visual Studio](../../extensibility/ux-guidelines/fonts-and-formatting-for-visual-studio.md).  
  
 Подробные сведения о написании пояснительного текста в разделе [UI text and terminology](../../extensibility/ux-guidelines/ui-text-and-help-for-visual-studio.md#BKMK_UITextAndTerminology).  
  
 ![Instructional text formatting](../../extensibility/ux-guidelines/media/070702-02_instructionaltextformatting.png "070702\-02\_InstructionalTextFormatting")  
  
 **Пояснительный текст в диалоговом окне Visual Studio**  
  
#### Информационный текст  
 Информационный текст — текст, который предоставляет пользователю Дополнительные сведения. Возможно, статические или динамические или используется для уведомления. Он всегда доступен только для чтения, но если это полезно для пользователя иметь возможность скопировать данные, динамический текст должно находиться в контейнере элемента управления, например только для чтения текстовом поле.  
  
##### Динамический текст \(контекстно зависимым\)  
 Текст динамической информации изменяется в зависимости от контекста, например при переключении фокуса. Часто, но не всегда динамическое содержимое объединяется с динамические метка.  
  
 ![Dynamic information text](../../extensibility/ux-guidelines/media/070702-03_informationaldynamictext.png "070702\-03\_InformationalDynamicText")  
  
 **Динамический информационный текст изменяется в зависимости от контекста.**  
  
##### Форматирование  
 Существует два способа отображения полей только для чтения текста: либо непосредственно в пользовательском Интерфейсе области \(см. выше\) или внутри другого элемента управления, например группы рамку или текстовое поле. Либо является правильным, в зависимости от ситуации. Возлагается конструктор компонентов, чтобы определить способ представления сведений только для чтения.  
  
 Текст может быть только для чтения текстовом поле. Это обычно означает, что содержимое можно выделить и скопировать, несмотря на то, что не может быть изменен.  
  
 ![Informational text formatting for read&#45;only fields](../../extensibility/ux-guidelines/media/070702-04_informationalformatting.png "070702\-04\_InformationalFormatting")  
  
 **Форматирование обучающего текста для полей, предназначенных только для чтения**  
  
#### Водяные знаки  
 Текст может быть таким же, разница между водяные знаки и пояснительного текста — что водяных знаков заменяются содержимым при окна элемента управления и не является пустым и постоянно остается доступным для пояснительного текста.  
  
 Водяные знаки можно использовать при пустом окна или элемента управления. Они указывают, что необходимо сделать для заполнения области и может включать действия, чтобы открыть соответствующие окна, такие как источник перетаскивания.  
  
##### Визуальный стиль  
  
-   Водяные знаки по центру по горизонтали в пределах окна.  
  
-   Водяных знаков должно быть выровнены по центру, не по левому краю.  
  
-   Водяные знаки могут располагается в верхней части области или вертикально по центру. Если находится в верхней области, чтобы выделялся водяного знака должно быть достаточно места выше.  
  
-   Использование `Environment.GrayText` шрифта цвет маркера и стандартные среды. Гиперссылки следует использовать маркеры стандартной гиперссылка общих: `Environment.PanelHyperlink`, `Environment.PanelHyperlinkHover`, `Environment.PanelHyperlinkPressed`, и `Environment.PanelHyperlinkDisabled`.  
  
-   Водяные знаки не могут быть выбраны на заднем плане  
  
-   Если это возможно включите ссылки водяного знака, чтобы помочь пользователю начать работу.

 ![Watermark text in a tool window](../../extensibility/ux-guidelines/media/070702-06_watermark2.png "070702\-06\_Watermark2")

##  <a name="BKMK_ButtonsAndHyperlinks"></a> Кнопки и гиперссылки  
  
### Обзор  
 Элементы управления кнопок и ссылки \(гиперссылок\) должны следовать [базовое руководство Windows Desktop по гиперссылкам](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742406\(v=vs.85\).aspx) для использования Формулировка призыва размеров и интервалы.  
  
### Выбор между кнопками и ссылками  
 Традиционно использовали кнопки для действий и гиперссылки были зарезервированы для навигации. Кнопки могут использоваться во всех случаях, но роль ссылки была расширена в Visual Studio, чтобы кнопки и ссылки взаимозаменяемы более при некоторых условиях.  
  
 Когда следует использовать кнопки:  
  
-   Основные команды  
  
-   Отображение windows используется для ввода данных или выбора параметров, даже если они являются дополнительные команды  
  
-   Удалением или необратимым действий  
  
-   Кнопки обязательства в мастерах и потоки страницы  
  
 Избегайте кнопок в окнах инструментов, или если вам требуется более двух слов для метки. Ссылки может иметь больше меток.  
  
 При использовании ссылок:  
  
-   Переход к другому окну, документа или веб\-страницы  
  
-   Ситуации, в которых требуется больше времени, метки или короткое предложение для описания цель действия  
  
-   Где кнопка будет перегрузки пользовательского интерфейса, при условии, что действие не разрушением или необратимым тесной пробелы  
  
-   Рассматриваем дополнительные команды в ситуациях, где имеется много команд

### Стандартные кнопки  
  
#### Text  
 Следуйте указаниям, изложенным записи [UI text and terminology](../../extensibility/ux-guidelines/ui-text-and-help-for-visual-studio.md#BKMK_UITextAndTerminology).  
  
#### Визуальный стиль  
  
##### Стандартные диалоговые окна  
 Большинство кнопок в Visual Studio будет отображаться в стандартные диалоговые окна и не быть со стилем. Они должны отражать стандартный внешний вид кнопки, определяемый операционной системы.  
  
##### Тематические  
 В некоторых случаях кнопок, которые могут быть использованы в со стилем пользовательского интерфейса и необходимо соответствующим образом со стилем этих кнопок. В разделе [Диалоговые окна](../../extensibility/ux-guidelines/application-patterns-for-visual-studio.md#BKMK_Dialogs) сведения об элементах управления темой.  
  
### Специальные кнопки  
  
#### Обзор... кнопки  
 **\[Обзор...\]** кнопки используются в таблицы, диалоговых окон, окон инструментов и другие немодальные элементы пользовательского интерфейса. Они отображают выбора, которое помогает пользователю при заполнении значение в элемент управления. Существует два варианта этой кнопки, длинные и короткие.  
  
 ![Long &#91;Browse...&#93; button](../../extensibility/ux-guidelines/media/070703-04_browselong.gif "070703\-04\_BrowseLong")  
  
 **Долго \[Обзор...\] button**  
  
 ![Short ellipsis&#45;only &#91;Browse...&#93; button](../../extensibility/ux-guidelines/media/070703-05_browseshort.gif "070703\-05\_BrowseShort")  
  
 **Кнопка только с многоточием short \[...\]**  
  
 Когда кнопка только с многоточием короткий:  
  
-   При наличии более длинный **\[Обзор...\]** кнопку в диалоговом окне, например, если разрешить несколько полей для просмотра. Использовать короткие **\[...\]** кнопка для каждого предотвратить путаницу клавиши доступа, созданные в этой ситуации \(**& Обзор** и **B & выбрать** в то же диалоговое окно\).  
  
-   В диалоговом окне тесной или нет разумного места для размещения длинная кнопка.  
  
-   Если кнопка будет отображаться в элементе управления сетки.  
  
 Рекомендации по использованию кнопки:  
  
-   Не использовать ключ доступа. Для доступа к ней с помощью клавиатуры, пользователю необходимо с помощью клавиши tab возле элемента управления. Убедитесь в наличии последовательности табуляции таким образом, что любой кнопки обзора находится сразу после поля, оно будет заполнено. Никогда не используйте символ подчеркивания ниже первого периода.  
  
-   Установите Microsoft Active Accessibility \(MSAA\) **имя** свойства **Обзор...** \(включая многоточие\), который экрана читателей прочтет его как «Обзор» и не «точка точка точка» или «период периодам.» Для управляемых элементов управления, это значит, параметр **AccessibleName** свойство.  
  
-   Никогда не используйте многоточие **\[...\]** кнопку ничего кроме действие просмотра. Например, если требуется **\[новое\]** кнопки, но не было достаточно места для текста, а затем диалогового окна должен быть изменено.

### Гиперссылки  
 Гиперссылки хорошо подходят для действия на основе переходов, например открытие раздела справки, модальное диалоговое окно или мастер. При использовании гиперссылки для команды она всегда отображают видимым и существенного изменения пользовательского интерфейса. Как правило действия, которые зафиксировать действие \(например, сохранение, Отмена и удаление\) лучше передаются с помощью кнопки.  
  
#### Стиль написания  
 Выполните [руководство Windows Desktop для текста интерфейса пользователя](https://msdn.microsoft.com/en-us/library/windows/desktop/dn742478\(v=vs.85\).aspx). Не используйте «Узнайте больше о,» «Сообщить мне об» или «Помощь в это» выражения. Вместо этого фразу текст ссылки справки с точки зрения основной вопрос, ответы на которые содержатся в справке. Например «**как добавить сервер в проводник по серверам?**»  
  
#### Визуальный стиль  
  
-   Следует всегда использовать гиперссылки [Службы VSColor](../../extensibility/ux-guidelines/colors-and-styling-for-visual-studio.md#BKMK_TheVSColorService). Если гиперссылки стили не определены правильно, красным, когда активны или показывает цвет после посещений.  
  
-   Не включайте подчеркивание управлять лег состояние, если ссылка является фрагмент предложение в полное предложение, такие как в водяного знака.  
  
-   Подчеркивание не должны отображаться при наведении указателя мыши. Вместо этого обратной связи с пользователем, что ссылка является активным является незначительные изменения и курсор соответствующую ссылку.  
  
##  <a name="BKMK_TreeViews"></a> Представление в виде дерева  
  
### Обзор  
 В древовидном представлении предоставляют способ упорядочить сложных перечислены в родительско дочерние группы. Пользователь может развернуть или свернуть родительские группы для отображения или скрытия базового дочерних элементов. Каждый элемент в виде дерева можно выбрать для дальнейшего действия.  
  
 В этом разделе рассматриваются допустимого использования надлежащее проектирование и функциональные возможности представления в виде дерева.

###  <a name="BKMK_TreeViewVisualStyle"></a> Визуальный стиль  
  
#### Расширители  
 Элементы управления древовидного представления должны соответствовать расширения схемы, используемой Windows и Visual Studio. Каждый узел использует элемента управления expander для отображения или скрытия элементов базовой. С помощью элемента управления expander обеспечивает согласованность для пользователей, которые могут возникнуть различные в древовидном представлении в Windows и Visual Studio.  
  
 ![Correct tree view control](../../extensibility/ux-guidelines/media/070705-1_treeviewcorrect.png "070705\-1\_TreeViewCorrect")  
  
 **Правильно: правильный стиль дерева с помощью элемента управления expander**  
  
 ![Incorrect tree view nodes](../../extensibility/ux-guidelines/media/070705-2_treeviewincorrect1.png "070705\-2\_TreeViewIncorrect1")  
  
 **Неправильный: неправильная стиль дерева**  
  
#### Выбранное  
 При выборе узла в представлении в виде дерева, выделения следует развернуть на всю ширину элемента управления иерархического представления. Это помогает пользователям четкого определения того, какой элемент выбран их. Выбор цвета должны отражать текущей темы Visual Studio.  
  
 ![Correct tree view control](../../extensibility/ux-guidelines/media/070705-1_treeviewcorrect.png "070705\-1\_TreeViewCorrect")  
  
 **Правильно: выделение выбранного узла соответствует всю ширину элемента управления иерархического представления.**  
  
 ![Incorrect tree view highlighting](../../extensibility/ux-guidelines/media/070705-3_treeviewincorrect2.png "070705\-3\_TreeViewIncorrect2")  
  
 **Неправильный: выделение выбранного узла не соответствует всю ширину элемента управления иерархического представления.**  
  
#### Значки  
 Значки следует использовать только в элементы управления древовидного представления, если они помогают визуально определить различия между элементами. Как правило значки следует использовать только в разнородных списки, в которых значки несут информацию для различения типов элементов. В однородных список значков часто можно рассматривать как шума и его следует избегать. В этом случае значок группы \(родительский\) может передать тип элементов в нем. Если значок является динамическим и используется для указания состояния, будет исключением из этого правила.  
  
#### Полосы прокрутки  
 Полосы прокрутки всегда должен быть скрыт, если содержимое помещается в элементе управления иерархического представления. Это допустимо для полосы прокрутки для скрытых или полупрозрачным в окне с прокруткой и отображаются, когда фокус находится на окне с древовидным представлением или при наведении курсора мыши в дереве просмотра сам.  
  
 ![Tree view with scroll bars](../../extensibility/ux-guidelines/media/070705-4_scrollbars.png "070705\-4\_Scrollbars")  
  
 **И вертикальной и горизонтальной полосы прокрутки отображаются, так как содержимое превышен лимит элемента управления иерархического представления.**  
  
###  <a name="BKMK_TreeViewInteractions"></a> Взаимодействие  
  
#### Контекстные меню  
 Дерева может содержать параметры вложенного меню в контекстное меню. Как правило это происходит, когда пользователь щелкает правой кнопкой мыши элемент или нажата клавиша меню на клавиатуре Windows с выбранного элемента. Очень важно, что узел получает фокус и выбран. Это помогает пользователю определить, какой элемент принадлежит подменю.  
  
 ![Selected tree node and context menu](../../extensibility/ux-guidelines/media/070705-5_contextmenu.png "070705\-5\_ContextMenu")  
  
 **Элемент, который имеет создания прибыли фокуса меню контекста для уведомления пользователя, какой элемент выбран.**  
  
#### Клавиатура  
 Представление дерева должен предоставить возможность выбора элементов и развернуть или свернуть узлы с помощью клавиатуры. Это гарантирует, что навигации требованиям наших специальных возможностей.  
  
##### Дерево  
 Дерево элементов управления Visual Studio должен выполнить общие с помощью клавиатуры:  
  
-   **Стрелка вверх:** выберите элементы, перемещая их вверх по дереву  
  
-   **Стрелка вниз:** выберите элементы, перемещая вниз по дереву  
  
-   **Стрелка вправо:** разверните узел в дереве  
  
-   **Стрелка влево:** свернуть узел в дереве  
  
-   **Клавишу ВВОД:** инициировать, загрузка и выполнение выбранного элемента  
  
##### Trid \(представление в виде дерева и представление сетки\)  
 Элемент управления trid — сложный элемент управления, который содержит представление в виде дерева в сетке. Развертывание, Свертывание и навигации по дереву должен подчиняться те же команды клавиатуры, представление в виде дерева с учетом следующих отличий:  
  
-   **Стрелка вправо:** развернуть узел. После развертывания узла, его следует продолжить перемещение ближайшего столбец справа. В конце строки следует остановить навигации.  
  
-   **Вкладка:** осуществляется переход к ближайшей ячейке справа.  В конце строки навигации продолжается на следующую строку.  
  
-   **Shift \+ вкладку:** осуществляется переход к ближайшей ячейке слева.  В начале строки навигации продолжает крайней правой ячейки в предыдущей строке.

 **Элемент управления trid в Visual Studio**
