# <a name="sccget-function"></a>Функция SccGet
Эта функция извлекает копия одного или нескольких файлов для просмотра и компиляции, но не для изменения. В большинстве систем файлы помечаются как доступные только для чтения.  
  
## <a name="syntax"></a>Синтаксис  
  
```cpp  
SCCRTN SccGet(  
   LPVOID    pvContext,  
   HWND      hWnd,  
   LONG      nFiles,  
   LPCSTR*   lpFileNames,  
   LONG      fOptions,  
   LPCMDOPTS pvOptions  
);  
```  
  
#### <a name="parameters"></a>Параметры  
 pvContext  
 [in] Структура контекста подключаемый модуль системы управления версиями.  
  
 hWnd  
 [in] Дескриптор окна интегрированной среды разработки, подключаемый модуль системы управления версиями можно использовать в качестве родительского для все диалоговые окна, которые он предоставляет.  
  
 nFiles  
 [in] Число файлов, указанных в `lpFileNames` массива.  
  
 lpFileNames  
 [in] Массив полных имен файлов, которые требуется получить.  
  
 fOptions  
 [in] Команда флаги (`SCC_GET_ALL`, `SCC_GET_RECURSIVE`).  
  
 pvOptions  
 [in] Параметры для конкретного подключаемого модуля системы управления версиями.  
  
## <a name="return-value"></a>Возвращаемое значение  
 Реализация подключаемого модуля управления источника этой функции должен возвращать одно из следующих значений:  
  
|Значение|Описание|  
|-----------|-----------------|  
|SCC_OK|Успешное выполнение операции get.|  
|SCC_E_FILENOTCONTROLLED|Файл не существует в системе управления версиями.|  
|SCC_E_OPNOTSUPPORTED|Система управления версиями не поддерживает эту операцию.|  
|SCC_E_FILEISCHECKEDOUT|Не удается получить файл, который пользователь в данный момент извлечен.|  
|SCC_E_ACCESSFAILURE|Возникла проблема с доступом к системе управления версиями, возможно, из-за проблемы с сетью или конфликтов. Рекомендуется повторить операцию.|  
|SCC_E_NOSPECIFIEDVERSION|Задана недопустимая версия или даты и времени.|  
|SCC_E_NONSPECIFICERROR|Неспецифичную сбоя; файл не был синхронизирован.|  
|SCC_I_OPERATIONCANCELED|Операция отменена до завершения.|  
|SCC_E_NOTAUTHORIZED|Пользователь не авторизован для выполнения этой операции.|  
  
## <a name="remarks"></a>Примечания  
 Эта функция вызывается с количеством и массив имен файлов, которые требуется получить. Если интегрированная среда разработки передает флаг `SCC_GET_ALL`, это означает, что элементы в `lpFileNames` не файлы, а также каталоги и все файлы в системе управления версиями в заданной каталоги должны быть получены.  
  
 `SCC_GET_ALL` Флаг может сочетаться с `SCC_GET_RECURSIVE` для получения всех файлов в данной каталоги и также всех его подкаталогах.  
  
> [!NOTE]
>  `SCC_GET_RECURSIVE` никогда не должны передаваться без `SCC_GET_ALL`. Кроме того Обратите внимание, что каталоги C:\A и C:\A\B, и переданы в рекурсивном получить C:\A\B и всех его подкаталогах фактически извлекается дважды. Интегрированная среда разработки ответственность — и не система управления подключаемого модуля — чтобы убедиться в том, что повторяющиеся значения, такие как это обеспечена из массива.  
  
 Наконец, даже если системы управления версиями подключаемого модуля указан `SCC_CAP_GET_NOUI` флаг при инициализации, указывающее, что он не имеет пользовательского интерфейса для команды Get, эта функция по-прежнему может вызываться в интегрированной среде разработки для получения файлов. Флаг просто означает, что IDE не отображается пункт меню Get и что подключаемый модуль не должен предоставлять никаких элементов пользовательского интерфейса.  
  
## <a name="renaming-and-sccget"></a>Переименование и SccGet  
 Ситуации: пользователь извлекает файл, например, a.txt и выполняется его изменение. Перед возвратом a.txt, другой пользователь переименовывает a.txt в b.txt в базе данных управления версиями, извлекает b.txt, вносит некоторые изменения в файл и проверяет файле. Первый пользователь хочет изменения, внесенные программой второй учетной записи пользователя, поэтому первый пользователь переименовывает файл a.txt их локальной версии в b.txt и выполняет запрос get для файла. Однако локальный кэш, который отслеживает номера версий по-прежнему предполагается первой версии a.txt хранится локально и поэтому системы управления версиями не может устранить различия.  
  
 Чтобы решить эту проблему, в котором локальный кэш версии управления исходным кодом становятся несинхронизированными с исходной базой данных управления двумя способами:  
  
1.  Не разрешать переименование файла в базы данных управления версиями, который в данный момент извлечен.  
  
2.  Сделайте эквивалентом «удалить старый» следуют «добавить новый». Следующий запрошенный алгоритм — один из способов решения этой задачи.  
  
    1.  Вызовите [SccQueryChanges](../extensibility/sccquerychanges-function.md) функции для получения сведений о переименовании a.txt в b.txt в базе данных управления версиями.  
  
    2.  Переименуйте локальный a.txt в b.txt.  
  
    3.  Вызовите `SccGet` для a.txt и b.txt функцию.  
  
    4.  Поскольку a.txt не существует в базе данных управления версиями, локальная версия кэша удаляются отсутствует a.txt сведений о версии.  
  
    5.  Извлеченного файла b.txt объединяется с содержимым файла локального b.txt.  
  
    6.  Теперь могут быть возвращены b.txt обновленный файл.
